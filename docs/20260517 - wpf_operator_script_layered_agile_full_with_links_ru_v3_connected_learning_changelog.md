   В каждый основной урок добавлен блок «Логическая связка урока»: что уже есть, какая новая боль появилась, какой механизм WPF берём, почему именно сейчас и какой рабочий результат должен получиться.

4. **`Layout` больше не вводится как отдельный словарный термин.**  
   Он появляется после дерева элементов: XAML уже создал `Window`, `Grid`, `Button`, но ещё не ясно, где они окажутся на экране. Поэтому нужен механизм компоновки.

5. **`Binding` больше не вводится до практической необходимости.**  
   Сначала мы показываем ручной способ через code-behind. Потом объясняем, что при сценарии из многих шагов ручное присваивание `TextBlock.Text = ...` и ручное создание кнопок станет хрупким. Поэтому появляется Binding.

6. **`DataContext` привязан к первому реальному Binding.**  
   Теперь он объясняется как ответ на вопрос: если XAML написал `{Binding HeaderText}`, у какого объекта WPF должен искать `HeaderText`?

7. **`MVVM` вводится не как лозунг, а как следствие роста сложности.**  
   Сначала допускается простой code-behind, чтобы получить рабочий срез. Затем показывается, почему окно начинает знать слишком много: хранит данные, выбирает следующий шаг, подставляет имена, обновляет список. После этого вводится ViewModel.

8. **Термины будущих уроков помечены как анонс, а не как обязательная теория прямо сейчас.**  
   Например, `DependencyProperty`, `RoutedEvent`, `Style`, `ControlTemplate`, `Dispatcher` связаны с конкретными будущими задачами проекта: привязка, обработка событий, стилизация под скриншоты, отзывчивость UI.

9. **Добавлена инструкция против копипаста.**  
   В начале учебника усилен порядок оффлайн-работы: читать короткий блок, понять проблему, написать код вручную, запустить, специально сломать, исправить, ответить на вопросы и только потом идти дальше.

10. **Сохранён agile-подход.**  
    Учебник по-прежнему идёт не водопадом, а рабочими вертикальными срезами: каждый урок добавляет маленькую завершённую возможность и заканчивается запускаемым кодом.

# 1. Как в целом работает WPF: не набор терминов, а цепочка причин

## Коротко

WPF-приложение удобно понимать не как список терминов, а как последовательность вопросов, которые появляются один за другим.

```text
Как приложение запускается?
    ↓
Как WPF понимает, какое окно открыть?
    ↓
Как XAML превращается в реальные объекты?
    ↓
Как эти объекты становятся деревом элементов?
    ↓
Как дерево размещается на экране?
    ↓
Как пользовательские действия попадают в C#?
    ↓
Как данные из C# показываются в XAML?
    ↓
Как WPF понимает, у какого объекта брать данные?
    ↓
Как не превратить окно в свалку логики?
```

Из этих вопросов естественно появляются основные понятия WPF:

- `App.xaml` — откуда стартует WPF-приложение;
- `Window` — главное окно;
- `XAML` — описание объектов интерфейса;
- `InitializeComponent()` — загрузка XAML и создание элементов;
- дерево элементов — структура интерфейса;
- `Layout` — размещение элементов на экране;
- события и команды — реакция на действия пользователя;
- `Binding` — связь данных C# с XAML;
- `DataContext` — объект, у которого Binding ищет данные;
- `MVVM` — способ не смешивать внешний вид и логику.

В этом учебнике каждый термин вводится только тогда, когда он решает конкретную проблему проекта.

---

## 1.1. Сначала есть задача, а не технология

Мы пишем WPF-аналог тестового задания: интерактивный скрипт оператора.

Минимально приложение должно уметь:

```text
1. Запуститься.
2. Показать окно.
3. Показать вкладки.
4. Спросить имя оператора и имя клиента.
5. Подставить эти данные в текст сценария.
6. Показать первый шаг сценария.
7. По ответу показать следующий шаг.
8. Если ответ изменили в раннем шаге — убрать лишние поздние шаги.
```

Если смотреть на это как на WPF-задачу, сразу видно, что нам понадобятся разные механизмы:

```text
показать окно                  → Window
описать внешний вид             → XAML
разложить элементы по экрану    → Layout
сделать вкладки                 → TabControl
получить имя оператора          → диалоговое окно
хранить шаг сценария            → Model
получить сценарий               → Service
показывать текущие шаги         → ObservableCollection
связать данные и интерфейс      → Binding + DataContext
обрабатывать ответы             → Command
не смешивать всё в окне         → MVVM
```

Важно: мы не будем изучать все эти термины сразу глубоко. Сначала увидим проблему, потом возьмём минимальный механизм WPF, который её решает, и только потом углубимся.

## 1.2. Как запускается WPF-приложение

### Коротко

Когда пользователь запускает WPF-приложение, .NET создаёт объект приложения, WPF читает `App.xaml`, находит стартовое окно, создаёт его и показывает на экране.

### Подробно

Типичная цепочка запуска такая:

```text
Пользователь запускает .exe
    ↓
.NET Runtime загружает приложение
    ↓
создаётся объект App
    ↓
WPF читает App.xaml
    ↓
WPF находит StartupUri
    ↓
создаётся MainWindow
    ↓
в конструкторе MainWindow вызывается InitializeComponent()
    ↓
XAML превращается в объекты интерфейса
    ↓
окно показывается пользователю
```

`App.xaml` отвечает на вопрос: **какое окно открыть при старте приложения?**

Пример:

```xml
<Application x:Class="OperatorScriptWpf.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             StartupUri="MainWindow.xaml">
    <Application.Resources>
    </Application.Resources>
</Application>
```

Здесь важная часть:

```xml
StartupUri="MainWindow.xaml"
```

Она говорит WPF: стартовое окно приложения — `MainWindow.xaml`.

### Почему это важно для нашего проекта

Пока у нас одно главное окно. Позже появится стартовый диалог с двумя полями. Но главное окно всё равно останется центром приложения: в нём будут вкладки, интерактивный сценарий и текущие открытые шаги.

### Что пока не трогаем глубоко

Пока не изучаем жизненный цикл `Application`, обработку глобальных ошибок, ресурсы приложения и переопределение `OnStartup`. Это понадобится позже, когда приложение станет сложнее.

Microsoft Learn: WPF Application Management — https://learn.microsoft.com/en-us/dotnet/desktop/wpf/app-development/application-management-overview

## 1.3. XAML: почему интерфейс описывается отдельно от C#

### Коротко

XAML — это способ описать интерфейс декларативно: не «создай кнопку руками в C#», а «в окне должна быть кнопка с такими свойствами».

### Подробно

В обычном C# можно было бы написать так:

```csharp
var button = new Button();
button.Content = "Сохранить";
button.Width = 120;
```

Это рабочий код. Но если весь интерфейс создавать так, он быстро станет плохо читаемым.

WPF предлагает другой способ:

```xml
<Button Content="Сохранить" Width="120" />
```

Это XAML. Он описывает объект `Button` и значения его свойств.

Когда проект собирается и запускается, WPF на основе XAML создаёт реальные C#-объекты:

```text
<Button Content="Сохранить" />
    ↓
new Button { Content = "Сохранить" }
```

Это УПРОЩЕНИЕ: в реальности WPF использует инфраструктуру XAML-загрузки, сгенерированные файлы и BAML. Но как учебная модель для начала важно понять главное: XAML описывает объекты, а WPF создаёт эти объекты при запуске.

### Почему это важно для нашего проекта

Наш интерфейс будет состоять из:

- шапки;
- вкладок;
- стартового диалога;
- списка шагов сценария;
- кнопок ответов;
- текстовых блоков.

Если всё создавать вручную в C#, мы потеряем читаемость. Поэтому внешний вид держим в XAML, а поведение и данные — в C#.

Microsoft Learn: XAML overview — https://learn.microsoft.com/en-us/dotnet/desktop/wpf/xaml/

## 1.4. `InitializeComponent()`: где XAML соединяется с C#

### Коротко

`InitializeComponent()` — метод, который загружает XAML текущего окна и создаёт элементы интерфейса. Обычно ты его не пишешь сам, но почти всегда видишь в конструкторе окна.

### Подробно

В файле `MainWindow.xaml.cs` обычно есть код:

```csharp
public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();
    }
}
```

Здесь несколько важных конструкций:

- `MainWindow` — наш класс окна;
- `: Window` — наследование от базового окна WPF;
- `partial` — класс разделён на несколько частей;
- `InitializeComponent()` — метод, который связывает C#-часть окна с XAML-частью.

Почему класс `partial`?

Потому что часть класса пишешь ты:

```text
MainWindow.xaml.cs
```

А часть класса генерируется инструментами WPF на основе:

```text
MainWindow.xaml
```

### Почему это важно для нашего проекта

Когда мы добавим в XAML кнопку с именем:

```xml
<Button x:Name="SaveButton" />
```

после `InitializeComponent()` C#-код сможет обращаться к ней как к объекту. Но в финальном варианте мы будем стараться меньше обращаться к элементам напрямую и больше использовать `Binding`.

### Что пока не трогаем глубоко

Пока не изучаем сгенерированные `.g.cs`-файлы подробно. Достаточно знать: если `InitializeComponent()` не вызван, XAML-элементы окна не будут созданы.

## 1.5. Дерево элементов: из чего состоит интерфейс

### Коротко

Когда WPF загрузил XAML, он получил не плоский список кнопок, а дерево объектов: окно содержит контейнер, контейнер содержит текст, кнопки, вкладки и другие элементы.

### Подробно

Пример XAML:

```xml
<Window>
    <Grid>
        <StackPanel>
            <TextBlock Text="Привет" />
            <Button Content="ОК" />
        </StackPanel>
    </Grid>
</Window>
```

Дерево:

```text
Window
└── Grid
    └── StackPanel
        ├── TextBlock
        └── Button
```

Это дерево важно не само по себе. Оно объясняет сразу несколько будущих механизмов WPF:

```text
дерево элементов
    ↓
Layout понимает, кто внутри кого расположен
    ↓
DataContext может наследоваться от родителя к дочерним элементам
    ↓
ресурсы могут искаться вверх по дереву
    ↓
некоторые события могут идти по дереву вверх или вниз
```

То есть дерево — это не отдельная теория, а основа многих практических вещей.

### Почему это важно для нашего проекта

Во вкладке `Представление` у нас будет список открытых шагов. Каждый шаг будет содержать текст и кнопки ответов. Это естественно ложится в дерево:

```text
MainWindow
└── TabControl
    └── TabItem "Представление"
        └── ItemsControl VisibleSteps
            └── шаблон одного шага
                ├── TextBlock с текстом шага
                └── ItemsControl с ответами
                    └── Button ответа
```

Когда мы дойдём до `ItemsControl` и `DataTemplate`, эта схема станет рабочим кодом.

## 1.6. От дерева элементов к Layout

### Коротко

Дерево отвечает на вопрос: **какие элементы есть внутри окна?**  
`Layout` отвечает на другой вопрос: **где эти элементы будут стоять и сколько места займут?**

### Подробно

После загрузки XAML WPF знает, что в окне есть `Grid`, внутри него `Button`, `TextBlock`, `TabControl`. Но этого мало.

Нужно решить:

- где будет шапка;
- сколько места займут вкладки;
- что произойдёт, если окно растянуть;
- как сделать, чтобы список шагов занимал свободное место;
- как расположить кнопки ответов.

Эти задачи решает система компоновки — `Layout`.

Упрощённая модель работы layout:

```text
1. Родительский элемент спрашивает дочерние: какой размер вам нужен?
2. Дочерние элементы сообщают желаемый размер.
3. Родитель решает, кому сколько места дать.
4. WPF размещает элементы в итоговых прямоугольниках.
5. Элементы рисуются на экране.
```

Это УПРОЩЕНИЕ. В WPF есть два основных прохода layout: measure — измерение и arrange — размещение. Для начала достаточно понимать смысл: WPF сначала выясняет размеры, потом раскладывает элементы.

Основные контейнеры:

- `Grid` — раскладка по строкам и колонкам;
- `StackPanel` — элементы друг за другом;
- `DockPanel` — элементы прижимаются к сторонам;
- `WrapPanel` — элементы переносятся на новую строку;
- `Canvas` — ручные координаты.

### Почему `Layout` появляется именно здесь

Потому что мы уже знаем: XAML создаёт дерево элементов. Следующий естественный вопрос: как дерево становится видимым расположением на экране?

В нашем проекте `Layout` нужен сразу:

```text
верхняя шапка приложения
    ↓
ниже область вкладок
    ↓
внутри вкладки список шагов
    ↓
внутри шага текст и кнопки ответов
```

Поэтому в ранних уроках мы используем `Grid`, а позже — `StackPanel`, `ItemsControl` и шаблоны.

Microsoft Learn: Layout — https://learn.microsoft.com/en-us/dotnet/desktop/wpf/advanced/layout

## 1.7. Первый способ реакции на действия: события и code-behind

### Коротко

Когда пользователь нажимает кнопку, WPF должен передать это действие в C#-код. Самый простой способ — событие `Click` в code-behind.

### Подробно

Пример:

```xml
<Button Content="Нажми меня" Click="Button_Click" />
```

C#:

```csharp
private void Button_Click(object sender, RoutedEventArgs e)
{
    MessageBox.Show("Кнопка нажата");
}
```

Это понятно и нормально для первого рабочего среза. Мы видим кнопку, нажимаем её, получаем результат.

Но если так писать весь проект, окно быстро начнёт делать слишком много:

```text
MainWindow.xaml.cs
    ├── хранит имя оператора
    ├── хранит имя клиента
    ├── хранит все шаги сценария
    ├── выбирает следующий шаг
    ├── удаляет лишние шаги
    ├── создаёт кнопки
    ├── обновляет тексты
    └── отвечает за внешний вид
```

Это плохой знак. Окно должно в основном отвечать за отображение, а не за всю бизнес-логику.

### Почему мы всё равно начинаем с code-behind

Потому что agile-подход требует сначала получить рабочий срез. Для начинающего важно увидеть:

```text
кнопка → обработчик → результат
```

После этого легче понять, почему нам понадобятся `Binding`, `Command` и `ViewModel`.

## 1.8. От ручного обновления UI к Binding

### Коротко

Сначала можно вручную менять текст в элементах интерфейса. Но для интерактивного сценария это быстро становится неудобно. `Binding` решает проблему связи данных C# и XAML.

### Подробно

Ручной способ:

```csharp
HeaderTextBlock.Text = "Интерактивный скрипт";
OperatorTextBlock.Text = operatorName;
```

Для двух элементов это нормально. Но наш проект не состоит из двух элементов. У нас будут:

- имя оператора;
- имя клиента;
- список открытых шагов;
- текст каждого шага;
- список ответов у каждого шага;
- состояние выбранных ответов;
- вкладки с разным содержимым.

Если всё обновлять руками, код станет хрупким:

```text
данные изменились
    ↓
надо не забыть найти нужный TextBlock
    ↓
надо вручную поменять Text
    ↓
надо вручную добавить или удалить элементы
    ↓
легко ошибиться и оставить UI в старом состоянии
```

WPF предлагает другой подход:

```xml
<TextBlock Text="{Binding HeaderText}" />
```

Это означает:

```text
WPF, возьми значение свойства HeaderText из объекта-источника данных
и покажи его в свойстве Text этого TextBlock.
```

`Binding` — это не магическое слово. Это договор между XAML и C#:

```text
XAML говорит: мне нужно свойство HeaderText.
C#-объект предоставляет: public string HeaderText { get; }
WPF соединяет их.
```

### Почему Binding появляется именно после code-behind

Пока у нас одна кнопка — code-behind достаточно. Когда появляются данные сценария и повторяющиеся шаги — ручное обновление UI мешает. Поэтому мы переходим к привязке данных.

Microsoft Learn: Data binding overview — https://learn.microsoft.com/en-us/dotnet/desktop/wpf/data/

## 1.9. Binding задаёт вопрос: где источник? DataContext отвечает

### Коротко

`Binding` говорит: «покажи свойство `HeaderText`».  
`DataContext` говорит: «искать `HeaderText` нужно вот в этом объекте».

### Подробно

Допустим, в XAML написано:

```xml
<TextBlock Text="{Binding HeaderText}" />
```

Возникает естественный вопрос:

```text
У какого объекта WPF должен искать HeaderText?
```

Ответ: у объекта, назначенного в `DataContext`.

Пример:

```csharp
DataContext = new MainViewModel();
```

Теперь WPF будет искать `HeaderText` в объекте `MainViewModel`:

```csharp
public sealed class MainViewModel
{
    public string HeaderText { get; } = "Интерактивный скрипт оператора";
}
```

Связка целиком:

```text
MainWindow.DataContext = MainViewModel
    ↓
<TextBlock Text="{Binding HeaderText}" />
    ↓
WPF ищет HeaderText в MainViewModel
    ↓
значение попадает в TextBlock.Text
```

### Почему DataContext появляется не раньше

Пока мы не используем `Binding`, `DataContext` почти не нужен. Он становится понятен именно в момент первого реального Binding. Поэтому в уроках мы сначала покажем ручной способ, потом Binding, а потом объясним `DataContext` на рабочем примере.

Microsoft Learn: Binding declarations overview — https://learn.microsoft.com/en-us/dotnet/desktop/wpf/data/binding-declarations-overview

## 1.10. Когда code-behind начинает мешать: путь к ViewModel

### Коротко

`ViewModel` появляется не потому, что «так модно», а потому что окно начинает знать слишком много. ViewModel забирает состояние экрана и действия пользователя из `MainWindow.xaml.cs`.

### Подробно

Представим, что мы продолжаем писать всё в `MainWindow.xaml.cs`.

Сначала там только:

```text
InitializeComponent()
```

Потом добавили:

```text
кнопка запуска
```

Потом:

```text
имя оператора
имя клиента
```

Потом:

```text
список шагов сценария
```

Потом:

```text
логика выбора ответа
логика удаления лишних шагов
логика подстановки имён
```

В какой-то момент `MainWindow.xaml.cs` превращается в смесь всего:

```text
внешний вид + данные + сценарий + бизнес-правила + обработка кнопок
```

Это плохо, потому что:

- код трудно читать;
- трудно тестировать логику без запуска окна;
- изменение сценария может сломать UI;
- изменение UI может затронуть бизнес-логику;
- сложнее развивать проект.

Решение: перенести состояние и действия экрана в отдельный класс — `MainViewModel`.

```text
MainWindow.xaml       → внешний вид
MainWindow.xaml.cs    → минимальная настройка окна
MainViewModel.cs      → состояние экрана и команды
ScriptService.cs      → получение сценария
ScriptStep.cs         → модель шага
ScriptAnswer.cs       → модель ответа
```

### Где здесь MVVM

Когда мы разделяем:

- `View` — окно и XAML;
- `ViewModel` — состояние экрана и команды;
- `Model` — данные сценария;

мы приходим к паттерну MVVM.

MVVM — не цель сама по себе. Это способ сделать WPF-проект управляемым.

Microsoft Learn описывает ViewModel как слой, который предоставляет свойства и команды для привязки View и уведомляет View об изменениях состояния. Эта идея напрямую применяется в нашем `MainViewModel`.

Microsoft Learn: MVVM — https://learn.microsoft.com/en-us/dotnet/architecture/maui/mvvm

---

## 1.11. Почему WPF особенно хорошо подходит для MVVM

WPF исторически хорошо сочетается с MVVM, потому что в нём есть встроенные механизмы:

```text
Binding
    ↓
связывает XAML со свойствами ViewModel

DataContext
    ↓
указывает, где искать свойства для Binding

INotifyPropertyChanged
    ↓
сообщает UI, что свойство изменилось

ObservableCollection
    ↓
сообщает UI, что в коллекции добавился или удалился элемент

ICommand
    ↓
позволяет кнопкам вызывать действия ViewModel без Click-обработчиков
```

Эти механизмы мы будем вводить постепенно.

Важно: в первом уроке не нужно понимать весь MVVM. Нужно понять траекторию:

```text
сначала делаем рабочий UI
    ↓
потом замечаем повторяющийся ручной код
    ↓
потом переносим данные в ViewModel
    ↓
потом связываем View и ViewModel через Binding
    ↓
потом кнопки переводим на Command
```

```text
Не так:
"Binding — это привязка данных. Запомни."

Так:
"Мы устали вручную менять TextBlock.Text. Нужен механизм, который сам берёт данные из C# и показывает их в XAML. Этот механизм называется Binding."
```

```text
    ↓
Берём ItemsControl, потому что он умеет показывать коллекцию.
```

# Урок 1. Минимальное WPF-приложение: окно запускается

## Логическая связка урока

### Что мы уже знаем

Из общей главы мы знаем: WPF-приложение стартует через `App.xaml`, открывает `MainWindow.xaml`, вызывает `InitializeComponent()` и создаёт дерево элементов из XAML.

### Какая проблема сейчас

Пока у нас нет даже минимального рабочего окна. Нельзя обсуждать `Binding`, `MVVM`, сервисы и команды, если приложение ещё не запускается.

### Что берём из WPF

Берём самый маленький набор:

- `Window` — главное окно;
- `XAML` — внешний вид окна;
- `MainWindow.xaml.cs` — C#-часть окна;
- `InitializeComponent()` — загрузка XAML.

### Почему не вводим больше

`Layout`, `Binding`, `DataContext`, `MVVM` пока не нужны. Если в первом уроке дать их глубоко, это будет терминологический шум. Сейчас цель — увидеть живое окно и понять стартовую цепочку.

### Рабочий результат урока

После урока приложение должно запускаться командой `dotnet run` и показывать простое окно.


## Коротко

Цель урока — получить первое запускаемое WPF-окно и понять, какие файлы создаёт шаблон.

Рабочий результат урока: приложение запускается и показывает пустое окно `MainWindow`.

## Теория по Microsoft Learn простыми словами

Официальный туториал Microsoft по созданию WPF-приложения начинается с выбора шаблона `WPF Application`. Шаблон создаёт `.csproj`, `App.xaml`, `App.xaml.cs`, `MainWindow.xaml`, `MainWindow.xaml.cs`.

Это минимальный набор:

- `.csproj` говорит .NET SDK, как собирать проект;

# Урок 4. Подключаем диалог к главному окну

## Логическая связка урока

### Что уже работает

Диалог умеет получать текст из двух полей.

### Какая новая проблема появилась

Диалог сам по себе бесполезен, пока главное окно не получит введённые значения и не сможет использовать их в сценарии.

### Что берём из WPF и C#

Используем `ShowDialog()` и свойства диалогового окна, через которые главное окно забирает результат.

### Почему это всё ещё не MVVM

### Рабочий результат урока

После урока главное окно должно знать имя оператора и клиента после закрытия диалога.


## Коротко

Теперь главное окно при загрузке показывает `StartupDialog`, получает данные и временно выводит их в заголовок.

Рабочий результат: приложение запускается, показывает диалог, после сохранения главное окно знает имя оператора и клиента.

## Теория WPF

У окна есть событие `Loaded`. Оно срабатывает, когда окно уже создано, XAML загружен, элементы готовы.

События — это простой способ реагировать на действия или жизненный цикл UI. Позже для пользовательских действий мы будем использовать `ICommand`, но для показа стартового диалога событие `Loaded` допустимо.

## Рабочий срез кода

### Изменение в `MainWindow.xaml`

Добавь атрибут:

```xml

```

Полная первая строка окна:

```xml
<Window x:Class="OperatorScriptWpf.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Title="Интерактивный скрипт оператора"
        Height="720"
        Width="1100"
        MinHeight="600"
        MinWidth="900"
        Loaded="MainWindow_Loaded">
```

### `MainWindow.xaml.cs`

```csharp
using System.Windows;

namespace OperatorScriptWpf;

public partial class MainWindow : Window
{
    private bool _startupDialogWasShown;

    public MainWindow()
    {
        InitializeComponent();
    }

    private void MainWindow_Loaded(object sender, RoutedEventArgs e)
    {
        if (_startupDialogWasShown)
            return;

        _startupDialogWasShown = true;

        var dialog = new StartupDialog
        {
            Owner = this
        };

        bool? result = dialog.ShowDialog();

        if (result == true)
        {
            Title = $"Интерактивный скрипт оператора — {dialog.OperatorName} / {dialog.ClientName}";
        }
    }
}
```

## Объяснение конструкций

`private bool _startupDialogWasShown;` — поле, которое хранит, был ли уже показан диалог. Это защита от повторного открытия.

`if (_startupDialogWasShown) return;` — если диалог уже показывали, выходим из метода.

`var dialog = new StartupDialog { Owner = this };` — создаём окно диалога и указываем владельца. `this` здесь означает текущий объект `MainWindow`.

`bool? result` — nullable `bool`. Метод `ShowDialog()` возвращает `true`, `false` или `null`.

`$"...{dialog.OperatorName}..."` — интерполяция строк. Позволяет вставлять значения переменных прямо в строку.

# Урок 5. Первый интерактивный шаг без архитектуры

## Логическая связка урока

### Что уже работает

У нас есть окно, вкладки и данные оператора/клиента.

### Какая новая проблема появилась

Теперь нужно показать первый шаг сценария и дать пользователю выбрать ответ. Пока у нас нет моделей, сервисов и ViewModel.

### Что делаем намеренно просто

Пишем первый интерактивный шаг напрямую, без архитектурной чистоты. Это нужно, чтобы увидеть поведение: текст шага → кнопка ответа → следующий текст.

### Почему это не финальный подход

Если продолжить так, `MainWindow.xaml.cs` начнёт хранить весь сценарий и всю логику переходов. Это станет причиной для следующих уроков: модели, сервис, ViewModel.

### Рабочий результат урока

После урока во вкладке `Представление` должен появиться первый шаг сценария и хотя бы одна кнопка ответа.

Сначала сделаем простую рабочую версию прямо в окне: текст приветствия и две кнопки `Да`/`Нет`. Это намеренно простой слой, чтобы увидеть механику.

Рабочий результат: во вкладке `Представление` появляется первый шаг, кнопки меняют текст.

## Теория WPF

На раннем этапе допустимо использовать code-behind, то есть файл `MainWindow.xaml.cs`. Но важно понимать: это учебный временный слой. Он нужен, чтобы быстро получить работающую механику.

Позже мы вынесем:

- данные сценария в `Models`;
- создание сценария в `Services`;
- состояние экрана в `ViewModels`;
- действия кнопок в `Commands`.

Это agile-подход: сначала проверяем идею, потом улучшаем дизайн.

# Урок 6. Модели: описываем сценарий данными

## Логическая связка урока

### Что уже работает

Первый сценарный шаг можно показать вручную.

### Какая новая проблема появилась

Ручной текст и ручные кнопки не масштабируются. Сценарий состоит из шагов и ответов, значит его нужно описать как данные.

### Что берём из C#

Создаём модели: `ScriptStepDefinition` и `ScriptAnswerDefinition`.

### Почему это слой Model

Модель не знает про WPF, кнопки, вкладки и окна. Она описывает предметную область: шаг сценария, текст, варианты ответов, идентификатор следующего шага.

### Рабочий результат урока

После урока сценарий должен быть представлен объектами C#, а не россыпью строк в обработчиках кнопок.


## Коротко

Теперь убираем захардкоженную логику из кнопок и описываем сценарий через классы-модели.

Рабочий результат: проект всё ещё запускается, но у нас появляется слой `Models`.

## Теория WPF и C#

WPF показывает интерфейс, но сам сценарий — это не интерфейс. Сценарий является данными:

- шаг;
- текст шага;
- список ответов;
- ссылка на следующий шаг.

Поэтому нужно создать модели. Модель — это класс, который описывает данные предметной области.

## Рабочий срез кода

### `Models/ScriptAnswer.cs`

```csharp

{
    public string Text { get; init; } = "";
    public string? NextStepId { get; init; }
}
```
### `Models/ScriptStepDefinition.cs`

```csharp
namespace OperatorScriptWpf.Models;

public sealed class ScriptStepDefinition
{
    public string Id { get; init; } = "";
    public string Theme { get; init; } = "";
    public string TextTemplate { get; init; } = "";
    public IReadOnlyList<ScriptAnswer> Answers { get; init; } = Array.Empty<ScriptAnswer>();
}
```

## Объяснение конструкций

`namespace OperatorScriptWpf.Models;` — пространство имён. Помогает группировать классы.

`public sealed class ScriptAnswer` — публичный класс ответа. `sealed` означает, что от него нельзя наследоваться. Для простых моделей это нормально.

`string? NextStepId` — строка может быть `null`. Это нужно для финального ответа, после которого нет следующего шага.

`init` — свойство можно задать при создании объекта, но нельзя менять потом обычным присваиванием. Это удобно для описания неизменяемых данных.

`IReadOnlyList<ScriptAnswer>` — список только для чтения. Мы говорим: у шага есть ответы, но внешний код не должен произвольно менять коллекцию.

`Array.Empty<ScriptAnswer>()` — пустой массив без лишнего выделения памяти. Это лучше, чем `null`, потому что код может спокойно обходить список ответов.

# Урок 9. MainViewModel: состояние экрана отдельно от окна

## Логическая связка урока

### Что уже работает

### Какая новая проблема появилась

После урока основное состояние экрана должно находиться в `MainViewModel`, а окно должно быть тоньше.

## Теория MVVM

`ViewModel` — модель представления. Она знает, что нужно показать на экране, но не должна знать, как именно это нарисовано в XAML.

В нашей ViewModel будет:

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

namespace OperatorScriptWpf.ViewModels;

public abstract class ObservableObject : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }

    protected bool SetProperty<T>(ref T field, T value, [CallerMemberName] string? propertyName = null)
    {
        if (EqualityComparer<T>.Default.Equals(field, value))
            return false;

        field = value;
        OnPropertyChanged(propertyName);
        return true;
    }
}
```
### `ViewModels/MainViewModel.cs`

```csharp
using System.Collections.ObjectModel;
using OperatorScriptWpf.Models;
using OperatorScriptWpf.Services;

namespace OperatorScriptWpf.ViewModels;

public sealed class MainViewModel : ObservableObject
{
    private readonly IReadOnlyDictionary<string, ScriptStepDefinition> _steps;
    private readonly string _operatorName;
    private readonly string _clientName;

    public MainViewModel(string operatorName, string clientName)
    {
        _operatorName = string.IsNullOrWhiteSpace(operatorName) ? "Оператор" : operatorName.Trim();
        _clientName = string.IsNullOrWhiteSpace(clientName) ? "Клиент" : clientName.Trim();

        var scriptService = new ScriptService();
        _steps = scriptService.LoadPresentationScript();

        AddStep("start");
    }

    public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();

    private void SelectAnswer(VisibleStepViewModel currentStep, ScriptAnswer answer)
    {
        int currentIndex = VisibleSteps.IndexOf(currentStep);

        if (currentIndex < 0)
            return;

        while (VisibleSteps.Count > currentIndex + 1)
        {
            VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
        }

        if (string.IsNullOrWhiteSpace(answer.NextStepId))
            return;

        AddStep(answer.NextStepId);
    }

    private void AddStep(string stepId)
    {
        if (!_steps.TryGetValue(stepId, out var definition))
            return;

        var visibleStep = new VisibleStepViewModel(
            definition.Id,
            definition.Theme,
            ApplyVariables(definition.TextTemplate));

        foreach (var answer in definition.Answers)
        {
            visibleStep.Answers.Add(
                new AnswerOptionViewModel(
                    answer.Text,
                    new RelayCommand(_ => SelectAnswer(visibleStep, answer))));
        }

        VisibleSteps.Add(visibleStep);
    }

    private string ApplyVariables(string template)
    {
        return template
            .Replace("{OperatorName}", _operatorName)
            .Replace("{ClientName}", _clientName);
    }
}
```

## Объяснение конструкций

`ObservableObject` реализует `INotifyPropertyChanged`. Этот интерфейс нужен, чтобы WPF узнавал об изменении свойств.

`PropertyChanged?.Invoke(...)` — вызвать событие, если на него кто-то подписан. `?.` защищает от `null`.

`[CallerMemberName]` — атрибут, который автоматически подставляет имя свойства, вызвавшего метод.

`private readonly` — поле можно присвоить только в конструкторе. После этого ссылка не меняется.

`string.IsNullOrWhiteSpace(operatorName)` — проверяет, что строка не `null`, не пустая и не состоит только из пробелов.

`Trim()` — удаляет пробелы в начале и конце строки.

`VisibleSteps.IndexOf(currentStep)` — ищем номер текущего шага в списке отображаемых шагов.

Цикл:

```csharp
while (VisibleSteps.Count > currentIndex + 1)
{
    VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
}
```

Удаляет все шаги после текущего. Это реализация требования ТЗ: если нажали ответ в раннем шаге, лишние последующие шаги скрываются.

`TryGetValue(stepId, out var definition)` — безопасно ищет шаг в словаре. Если шаг есть, переменная `definition` получает значение.

# Урок 10. RelayCommand и ICommand: кнопки без Click-обработчиков

## Логическая связка урока

### Что уже работает

Состояние экрана перенесено во ViewModel.

### Какая новая проблема появилась

Кнопки в XAML всё ещё могут быть привязаны к `Click`-обработчикам в окне. Это возвращает логику действий обратно во View.

### Что берём из WPF

Берём `ICommand`: способ связать кнопку с действием ViewModel.

### Почему нужен `RelayCommand`

`ICommand` — интерфейс, то есть контракт. Нужна конкретная реализация, которая вызовет нужный метод. Для этого пишем простой `RelayCommand`.

### Рабочий результат урока

После урока кнопки должны вызывать действия ViewModel без `Click`-обработчиков в `MainWindow.xaml.cs`.


## Коротко

Заменяем обработчики `Click` на команды. Это важный шаг к нормальному WPF/MVVM.

Рабочий результат: кнопки ответов выполняют команды, которые добавляют следующий шаг.

## Теория WPF

`ICommand` — интерфейс команды. В WPF кнопка умеет выполнять команду через свойство `Command`.

Команда отвечает на два вопроса:

1. Можно ли выполнить действие сейчас? `CanExecute`
2. Что делать при выполнении? `Execute`

`RelayCommand` — универсальная команда-обёртка. Мы передаём ей делегат, а она реализует `ICommand`.

## Рабочий срез кода

### `ViewModels/RelayCommand.cs`

```csharp
using System.Windows.Input;

namespace OperatorScriptWpf.ViewModels;

public sealed class RelayCommand : ICommand
{
    private readonly Action<object?> _execute;
    private readonly Predicate<object?>? _canExecute;

    public RelayCommand(Action<object?> execute, Predicate<object?>? canExecute = null)
    {
        _execute = execute ?? throw new ArgumentNullException(nameof(execute));
        _canExecute = canExecute;
    }

    public bool CanExecute(object? parameter)
    {
        return _canExecute?.Invoke(parameter) ?? true;
    }

    public void Execute(object? parameter)
    {
        _execute(parameter);
    }

    public event EventHandler? CanExecuteChanged;

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

## Объяснение конструкций

`Action<object?> _execute` — действие, которое надо выполнить.

`Predicate<object?>? _canExecute` — условие, можно ли выполнить команду. `Predicate` возвращает `true` или `false`.

`execute ?? throw new ArgumentNullException(nameof(execute))` — если `execute` равен `null`, выбросить понятную ошибку.

`_canExecute?.Invoke(parameter) ?? true` — если условие задано, вызываем его; если не задано, считаем команду доступной.

`CanExecuteChanged` — событие, через которое команда сообщает WPF: доступность команды изменилась.

# Урок 11. Binding в XAML: View подключается к ViewModel

## Объяснение конструкций

`ItemsSource="{Binding VisibleSteps}"` — привязываем список элементов интерфейса к коллекции `VisibleSteps` из `MainViewModel`.

`DataTemplate` — говорит WPF, как отрисовать один `VisibleStepViewModel`.

`Text="{Binding Text}"` — берём свойство `Text` текущего шага.

Внутренний `ItemsControl ItemsSource="{Binding Answers}"` показывает ответы текущего шага.

`Command="{Binding SelectCommand}"` — кнопка ответа выполняет команду, которая была создана во ViewModel.

`ScrollViewer` добавляет прокрутку, если шагов стало много.

`DataContext = new MainViewModel(...)` — главное соединение View и ViewModel.

```csharp
using System.Collections.ObjectModel;
using OperatorScriptWpf.Models;
using OperatorScriptWpf.Services;

namespace OperatorScriptWpf.ViewModels;

public sealed class MainViewModel : ObservableObject
{
    private readonly IReadOnlyDictionary<string, ScriptStepDefinition> _steps;
    private readonly string _operatorName;
    private readonly string _clientName;

    public MainViewModel(string operatorName, string clientName)
    {
        _operatorName = string.IsNullOrWhiteSpace(operatorName) ? "Оператор" : operatorName.Trim();
        _clientName = string.IsNullOrWhiteSpace(clientName) ? "Клиент" : clientName.Trim();

        var scriptService = new ScriptService();
        _steps = scriptService.LoadPresentationScript();

        AddStep("start");
    }

    public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();

    private void SelectAnswer(VisibleStepViewModel currentStep, ScriptAnswer answer)
    {
        int currentIndex = VisibleSteps.IndexOf(currentStep);

        if (currentIndex < 0)
            return;

        while (VisibleSteps.Count > currentIndex + 1)
        {
            VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
        }

        if (string.IsNullOrWhiteSpace(answer.NextStepId))
            return;

        AddStep(answer.NextStepId);
    }

    private void AddStep(string stepId)
    {
        if (!_steps.TryGetValue(stepId, out var definition))
            return;

        var visibleStep = new VisibleStepViewModel(
            definition.Id,
            definition.Theme,
            ApplyVariables(definition.TextTemplate));

        foreach (var answer in definition.Answers)
        {
            visibleStep.Answers.Add(
                new AnswerOptionViewModel(
                    answer.Text,
                    new RelayCommand(_ => SelectAnswer(visibleStep, answer))));
        }

        VisibleSteps.Add(visibleStep);
    }

    private string ApplyVariables(string template)
    {
        return template
            .Replace("{OperatorName}", _operatorName)
            .Replace("{ClientName}", _clientName);
    }
}
```

```csharp
namespace OperatorScriptWpf.ViewModels;

public abstract class ObservableObject : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }

    protected bool SetProperty<T>(ref T field, T value, [CallerMemberName] string? propertyName = null)
    {
        if (EqualityComparer<T>.Default.Equals(field, value))
            return false;

        field = value;
        OnPropertyChanged(propertyName);
        return true;
    }
}
```

```csharp
using System.Windows.Input;

namespace OperatorScriptWpf.ViewModels;

public sealed class RelayCommand : ICommand
{
    private readonly Action<object?> _execute;
    private readonly Predicate<object?>? _canExecute;

    public RelayCommand(Action<object?> execute, Predicate<object?>? canExecute = null)
    {
        _execute = execute ?? throw new ArgumentNullException(nameof(execute));
        _canExecute = canExecute;
    }

    public bool CanExecute(object? parameter)
    {
        return _canExecute?.Invoke(parameter) ?? true;
    }

    public void Execute(object? parameter)
    {
        _execute(parameter);
    }

    public event EventHandler? CanExecuteChanged;

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

```csharp
public sealed class ScriptAnswer
```

### Подробнее

Разберём построчно.

```csharp
namespace OperatorScriptWpf.Models;
```

`namespace` — пространство имён.

Оно нужно, чтобы группировать классы и избегать конфликтов имён.

Например, в большом проекте может быть много классов `User`, `Order`, `Step`. Пространство имён помогает понять, к какой части проекта относится класс.

```text
OperatorScriptWpf.Models
```

означает:


Разберём слова.

`public` — публичный.  
Класс доступен из других частей проекта.

#### `sealed`

`sealed` — запечатанный.

Это означает, что от этого класса нельзя наследоваться.

То есть нельзя написать:

```csharp
public class MyAnswer : ScriptAnswer
```

В данном проекте `ScriptAnswer` — простая модель данных. Мы не планируем расширять её через наследование. `sealed` явно показывает это намерение.

Для начинающего это не критично. Можно было бы написать и так:

```csharp
public class ScriptAnswer
```

Но `sealed` часто используют для простых конечных классов.

#### `class`

`class` — класс.  
Класс описывает тип объекта.

`ScriptAnswer` — это тип данных для ответа.

---

```csharp
public string Text { get; init; } = "";
```

Это свойство.

#### `string`

`string` — строка.

#### `Text`

Имя свойства. Здесь хранится текст ответа, например:

```text

Нет
Продолжить
Отказ
```

Это свойство можно:

- читать через `get`;
- задать при создании объекта через `init`.

```csharp
var answer = new ScriptAnswer

};
```

То есть нельзя:

Это хорошо для модели сценария: ответ задан один раз и дальше не должен случайно меняться.

## 6. Модель шага `ScriptStepDefinition`

### Коротко

`ScriptStepDefinition` описывает один шаг сценария.

Код:

```csharp
namespace OperatorScriptWpf.Models;

public sealed class ScriptStepDefinition
{
    public string Id { get; init; } = "";
    public string Theme { get; init; } = "";
    public string TextTemplate { get; init; } = "";
    public IReadOnlyList<ScriptAnswer> Answers { get; init; } = Array.Empty<ScriptAnswer>();
}
```

---

### Подробнее

Шаг сценария — это блок, который оператор видит на экране.

У шага есть:

- `Id` — техническое имя шага;
- `Theme` — тема;
- `TextTemplate` — текст с шаблонами;
- `Answers` — список ответов.

---

```csharp
public string Id { get; init; } = "";
```

`Id` — идентификатор.

Примеры:

```text
start
ready
offer
price
success
reject
```

Идентификатор нужен программе, а не пользователю.

Когда ответ указывает:

```csharp
NextStepId = "ready"
```

программа ищет шаг:

```csharp
Id = "ready"
```

---

```csharp
public string Theme { get; init; } = "";
```

`Theme` — тема шага.

В исходном задании было сказано, что описание в столбце «Тема» нужно оставить пустым. Поэтому в учебном проекте это поле обычно пустое.

Но мы оставляем свойство `Theme`, потому что:

1. структура ближе к заданию;
2. в будущем можно заполнить тему;
3. интерфейс уже готов к этому.

---

```csharp
public string TextTemplate { get; init; } = "";
```

`TextTemplate` — шаблон текста.

Почему не просто `Text`?

Потому что внутри текста есть переменные:

```text
Здравствуйте, меня зовут {OperatorName}
Я разговариваю с {ClientName}?
```

`{OperatorName}` и `{ClientName}` — это placeholders.

**Placeholder** — заполнитель.  
Перевод: «место для значения».

Позже программа заменит:

```text
{OperatorName}
```

на реальное имя оператора.

И заменит:

```text
{ClientName}
```

на имя клиента.

---

```csharp
public IReadOnlyList<ScriptAnswer> Answers { get; init; } = Array.Empty<ScriptAnswer>();
```

Разберём сложную конструкцию.

#### `IReadOnlyList<ScriptAnswer>`

Это интерфейс списка только для чтения.

- `IReadOnlyList` — список, который можно читать;
- `<ScriptAnswer>` — список состоит из объектов `ScriptAnswer`.

То есть свойство `Answers` содержит варианты ответа.

Пример:

```csharp
Answers =
[
    new ScriptAnswer { Text = "Да", NextStepId = "ready" },
    new ScriptAnswer { Text = "Нет", NextStepId = "wrong-client" }
]
```

#### Почему `IReadOnlyList`, а не `List`?

Потому что определение сценария лучше не менять случайно после создания.

`List` позволяет добавлять и удалять элементы.

`IReadOnlyList` говорит:

> Я отдаю список для чтения. Не надо менять его снаружи.

Это не абсолютная защита, но хороший архитектурный сигнал.

#### `Array.Empty<ScriptAnswer>()`

Это пустой массив ответов.

Он нужен, если у шага нет ответов.

Например, финальный шаг:

```csharp
Answers = []
```

или по умолчанию:

```csharp
Array.Empty<ScriptAnswer>()
```

## 7. Сервис сценария `ScriptService`

### Коротко

`ScriptService` создаёт набор шагов и возвращает их в виде словаря.

Код:

```csharp
using OperatorScriptWpf.Models;

namespace OperatorScriptWpf.Services;

public sealed class ScriptService
{
    public IReadOnlyDictionary<string, ScriptStepDefinition> LoadPresentationScript()
    {
        var steps = new List<ScriptStepDefinition>
        {
            new()
            {
                Id = "start",
                Theme = "",
                TextTemplate = """
                Здравствуйте, меня зовут {OperatorName}, я представляю компанию.
                Я разговариваю с {ClientName}?
                """,
                Answers =
                [
                    new ScriptAnswer { Text = "Да", NextStepId = "ready" },
                    new ScriptAnswer { Text = "Нет", NextStepId = "wrong-client" }
                ]
            }
        };

        return steps.ToDictionary(x => x.Id);
    }
}
```

### Подробнее

Сервис отвечает за загрузку сценария.

Сейчас сценарий записан прямо в C#-коде. Это нормально для учебного проекта.

В боевом проекте сценарий лучше вынести:

- в JSON;
- в базу данных;
- в отдельный редактор сценариев.

Но для понимания WPF сначала проще держать его в C#.

---

```csharp
using OperatorScriptWpf.Models;
```

`using` подключает пространство имён.

Без этой строки пришлось бы писать полное имя класса:

```csharp
OperatorScriptWpf.Models.ScriptAnswer
```

С `using` можно писать коротко:

```csharp

```

```csharp
public IReadOnlyDictionary<string, ScriptStepDefinition> LoadPresentationScript()
```

Это метод.

Метод доступен снаружи класса.

## 8. Почему нужен `ObservableObject`

### Коротко

`ObservableObject` — базовый класс для ViewModel.

Он нужен, чтобы WPF узнавал об изменениях свойств.

Код:

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

namespace OperatorScriptWpf.ViewModels;

public abstract class ObservableObject : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }

    protected bool SetProperty<T>(ref T field, T value, [CallerMemberName] string? propertyName = null)
    {
        if (EqualityComparer<T>.Default.Equals(field, value))
            return false;

        field = value;
        OnPropertyChanged(propertyName);
        return true;
    }
}
```

---

### Подробнее

WPF часто работает через binding.

**Binding** — привязка данных.

Например, в XAML:

```xml
<TextBlock Text="{Binding StatusText}" />
```

Это значит:

> Возьми свойство `StatusText` из `DataContext` и покажи его в `TextBlock`.

Но если `StatusText` изменится в C#-коде, интерфейс должен узнать об этом.

Для этого используется интерфейс:

```csharp
INotifyPropertyChanged
```

Перевод:

- `Notify` — уведомлять;
- `Property` — свойство;
- `Changed` — изменилось.

То есть:

> Я умею уведомлять, что свойство изменилось.

---

```csharp
public abstract class ObservableObject : INotifyPropertyChanged
```

#### `abstract`

`abstract` — абстрактный.

Это значит, что нельзя создать объект напрямую:

```csharp
var obj = new ObservableObject(); // нельзя
```

Этот класс нужен как база для других ViewModel.

Например:

```csharp
public sealed class MainViewModel : ObservableObject
```

#### `: INotifyPropertyChanged`

Двоеточие означает наследование или реализацию интерфейса.

Здесь класс реализует интерфейс `INotifyPropertyChanged`.

---

```csharp
public event PropertyChangedEventHandler? PropertyChanged;
```

Это событие.

**Event** — событие.

Событие позволяет одному объекту сообщить другим:

> У меня что-то произошло.

В данном случае:

> У меня изменилось свойство.

`PropertyChangedEventHandler?` — тип обработчика события. `?` означает, что пока никто не подписался, событие может быть `null`.

---

```csharp
protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
```

Метод вызывает событие изменения свойства.

#### `protected`

`protected` означает, что метод доступен внутри этого класса и внутри наследников.

То есть `MainViewModel` сможет вызвать `OnPropertyChanged`, а внешний код — нет.

#### `[CallerMemberName]`

Это атрибут.

Он автоматически подставляет имя свойства или метода, который вызвал `OnPropertyChanged`.

Например:

```csharp
OnPropertyChanged();
```

внутри свойства `StatusText` автоматически передаст строку:

```text
"StatusText"
```

---

```csharp
PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
```

Разберём.

#### `PropertyChanged?`

Оператор `?.` называется null-conditional operator.

Перевод: «оператор условного доступа при null».

Он значит:

> Если `PropertyChanged` не `null`, вызови `Invoke`. Если `null`, ничего не делай.

Безопасная форма.

#### `Invoke`

Вызвать событие.

#### `this`

`this` — текущий объект.

#### `new PropertyChangedEventArgs(propertyName)`

Создаём объект с именем изменившегося свойства.

---

```csharp
protected bool SetProperty<T>(ref T field, T value, [CallerMemberName] string? propertyName = null)
```

Это универсальный метод для изменения свойства.

#### `<T>`

Это generic type parameter.

Перевод: «обобщённый параметр типа».

Проще: метод может работать с разными типами.

Например:

- `string`;
- `int`;
- `bool`;
- свои классы.

#### `ref T field`

`ref` означает передачу по ссылке.

Метод сможет изменить реальное поле, которое ему передали.

#### `T value`

Новое значение.

---

```csharp
if (EqualityComparer<T>.Default.Equals(field, value))
    return false;
```

Проверяем: старое значение равно новому?

Если да — ничего не меняем и возвращаем `false`.

Это полезно, чтобы не отправлять лишние уведомления интерфейсу.

---

```csharp
field = value;
OnPropertyChanged(propertyName);
return true;
```

Если значение действительно изменилось:

1. записываем новое значение;
2. уведомляем WPF;
3. возвращаем `true`.

## 9. Команды и `RelayCommand`

### Коротко

В WPF кнопки лучше связывать не только с `Click`, но и с командами.

`RelayCommand` позволяет кнопке вызвать C#-действие из ViewModel.

Код:

```csharp
using System.Windows.Input;

namespace OperatorScriptWpf.ViewModels;

public sealed class RelayCommand : ICommand
{
    private readonly Action<object?> _execute;
    private readonly Predicate<object?>? _canExecute;

    public RelayCommand(Action<object?> execute, Predicate<object?>? canExecute = null)
    {
        _execute = execute ?? throw new ArgumentNullException(nameof(execute));
        _canExecute = canExecute;
    }

    public bool CanExecute(object? parameter)
    {
        return _canExecute?.Invoke(parameter) ?? true;
    }

    public void Execute(object? parameter)
    {
        _execute(parameter);
    }

    public event EventHandler? CanExecuteChanged;

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

---

### Подробнее

В простом WPF можно писать так:

```xml
<Button Content="Сохранить" Click="SaveButton_Click" />
```

И обработчик:

```csharp
private void SaveButton_Click(object sender, RoutedEventArgs e)
{
    // логика
}
```

Для старта это нормально.

Но в MVVM лучше, чтобы кнопка была связана с командой:

```xml
<Button Content="Сохранить" Command="{Binding SaveCommand}" />
```

Тогда логика живёт во ViewModel, а не в окне.

---

```csharp
public sealed class RelayCommand : ICommand
```

`ICommand` — стандартный WPF-интерфейс команды.

Он требует реализовать:

- `CanExecute` — можно ли выполнить команду;
- `Execute` — выполнить команду;
- `CanExecuteChanged` — событие изменения доступности команды.

---

```csharp
private readonly Action<object?> _execute;
```

Это приватное поле.

#### `private`

Доступно только внутри класса.

#### `readonly`

Можно присвоить значение только в конструкторе.

После этого поле менять нельзя.

#### `Action<object?>`

`Action` — делегат, то есть ссылка на метод.

Проще:

> Это действие, которое ничего не возвращает.

`object?` означает, что действие может принять параметр любого типа или `null`.

---

```csharp
private readonly Predicate<object?>? _canExecute;
```

`Predicate` — делегат, который возвращает `bool`.

То есть это проверка:

```text
можно выполнить команду? да/нет
```

`?` означает, что проверка может отсутствовать.

Если проверки нет — команда считается доступной.

---

```csharp
public RelayCommand(Action<object?> execute, Predicate<object?>? canExecute = null)
```

Это конструктор.

**Constructor** — конструктор.  
Он вызывается при создании объекта.

Например:

```csharp
var command = new RelayCommand(_ => DoSomething());
```

---

```csharp
_execute = execute ?? throw new ArgumentNullException(nameof(execute));
```

Разберём.

#### `??`

Оператор null-coalescing.

Перевод: «оператор объединения с null».

Он означает:

> Если слева не null — используй левое. Если null — используй правое.

Здесь:

```csharp
execute ?? throw new ArgumentNullException(nameof(execute))
```

значит:

> Если `execute` не null, сохранить его. Если null, выбросить ошибку.

#### `throw new ArgumentNullException(...)`

Выбросить исключение, если обязательный аргумент не передали.

#### `nameof(execute)`

Возвращает строку:

```text
"execute"
```

Плюс `nameof` безопаснее, чем писать строку вручную. Если параметр переименовать, компилятор поможет обновить имя.

---

```csharp
public bool CanExecute(object? parameter)
{
    return _canExecute?.Invoke(parameter) ?? true;
}
```

Если `_canExecute` есть, вызываем его.

Если `_canExecute` нет, возвращаем `true`.

То есть команда по умолчанию доступна.

---

```csharp

{
    _execute(parameter);
}
```

Выполняем действие, которое передали в команду.

## 12. Главная ViewModel `MainViewModel`

### Коротко

`MainViewModel` — главный мозг экрана.

Она:

- хранит все шаги;
- хранит видимые шаги;
- подставляет имя оператора и клиента;
- добавляет следующий шаг;
- удаляет лишние шаги, если пользователь изменил ранний ответ.

Полный код:

```csharp
using System.Collections.ObjectModel;
using OperatorScriptWpf.Models;
using OperatorScriptWpf.Services;

namespace OperatorScriptWpf.ViewModels;

public sealed class MainViewModel : ObservableObject
{
    private readonly IReadOnlyDictionary<string, ScriptStepDefinition> _steps;
    private readonly string _operatorName;
    private readonly string _clientName;

    public MainViewModel(string operatorName, string clientName)
    {
        _operatorName = string.IsNullOrWhiteSpace(operatorName) ? "Оператор" : operatorName.Trim();
        _clientName = string.IsNullOrWhiteSpace(clientName) ? "Клиент" : clientName.Trim();

        var scriptService = new ScriptService();
        _steps = scriptService.LoadPresentationScript();

        AddStep("start");
    }

    public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();

    private void SelectAnswer(VisibleStepViewModel currentStep, ScriptAnswer answer)
    {
        int currentIndex = VisibleSteps.IndexOf(currentStep);

        if (currentIndex < 0)
            return;

        while (VisibleSteps.Count > currentIndex + 1)
        {
            VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
        }

        if (string.IsNullOrWhiteSpace(answer.NextStepId))
            return;

        AddStep(answer.NextStepId);
    }

    private void AddStep(string stepId)
    {
        if (!_steps.TryGetValue(stepId, out var definition))
            return;

        var visibleStep = new VisibleStepViewModel(
            definition.Id,
            definition.Theme,
            ApplyVariables(definition.TextTemplate));

        foreach (var answer in definition.Answers)
        {
            visibleStep.Answers.Add(
                new AnswerOptionViewModel(
                    answer.Text,
                    new RelayCommand(_ => SelectAnswer(visibleStep, answer))));
        }

        VisibleSteps.Add(visibleStep);
    }

    private string ApplyVariables(string template)
    {
        return template
            .Replace("{OperatorName}", _operatorName)
            .Replace("{ClientName}", _clientName);
    }
}
```

## 13. Главная логика выбора ответа

### Коротко

Метод `SelectAnswer` делает главное:

1. находит текущий шаг;
2. удаляет все шаги ниже него;
3. если у ответа есть следующий шаг — добавляет его.

Код:

```csharp
private void SelectAnswer(VisibleStepViewModel currentStep, ScriptAnswer answer)
{
    int currentIndex = VisibleSteps.IndexOf(currentStep);

    if (currentIndex < 0)
        return;

    while (VisibleSteps.Count > currentIndex + 1)
    {
        VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
    }

    if (string.IsNullOrWhiteSpace(answer.NextStepId))
        return;

    AddStep(answer.NextStepId);
}
```

---

### Подробнее

Представим, что на экране видны шаги:

```text
0. start
1. ready
2. offer
3. price
```

Пользователь возвращается к шагу `ready` и выбирает другой ответ.

Тогда шаги `offer` и `price` уже не должны быть видны, потому что они относятся к старой ветке сценария.

---

```csharp
int currentIndex = VisibleSteps.IndexOf(currentStep);
```

`IndexOf` ищет индекс текущего шага в коллекции.

Индекс — это номер элемента.

В C# индексы начинаются с нуля:

```text
0 — первый элемент
1 — второй элемент
2 — третий элемент
```

Если шаг найден, будет число `0`, `1`, `2` и т.д.

Если не найден, будет `-1`.

---

```csharp
if (currentIndex < 0)
    return;
```

Если шаг не найден, выходим из метода.

`return` здесь означает:

> прекратить выполнение метода.

Почему это нужно?

Это защитная проверка. В нормальной ситуации шаг должен быть найден, но хороший код не должен падать из-за неожиданной ситуации.

---

```csharp
while (VisibleSteps.Count > currentIndex + 1)
{
    VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
}
```

Это цикл.

**while** — пока.

Он выполняется, пока условие истинно.

Условие:

```csharp
VisibleSteps.Count > currentIndex + 1
```

означает:

> Пока в списке есть элементы после текущего шага.

Пример.

Есть 4 шага:

```text
Count = 4
```

Текущий индекс:

```text
currentIndex = 1
```

Нужно оставить элементы:

```text
0 и 1
```

То есть количество должно стать:

```text
currentIndex + 1 = 2
```

Пока `Count > 2`, удаляем последний элемент.

---

```csharp
VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
```

Удалить элемент по индексу.

`VisibleSteps.Count - 1` — индекс последнего элемента.

Почему минус 1?

Потому что индексы начинаются с нуля.

Если элементов 4, индексы такие:

```text
0, 1, 2, 3
```

Последний индекс:

```text
4 - 1 = 3
```

---

```csharp
if (string.IsNullOrWhiteSpace(answer.NextStepId))
    return;
```

Если у ответа нет следующего шага, выходим.

Это нормальная ситуация для финального шага.

---

```csharp
AddStep(answer.NextStepId);
```

Добавляем следующий шаг.

## 14. Метод добавления шага `AddStep`

### Коротко

`AddStep` берёт определение шага, превращает его в видимый шаг и добавляет на экран.

Код:

```csharp
private void AddStep(string stepId)
{
    if (!_steps.TryGetValue(stepId, out var definition))
        return;

    var visibleStep = new VisibleStepViewModel(
        definition.Id,
        definition.Theme,
        ApplyVariables(definition.TextTemplate));

    foreach (var answer in definition.Answers)
    {
        visibleStep.Answers.Add(
            new AnswerOptionViewModel(
                answer.Text,
                new RelayCommand(_ => SelectAnswer(visibleStep, answer))));
    }

    VisibleSteps.Add(visibleStep);
}
```

---

### Подробнее

```csharp
if (!_steps.TryGetValue(stepId, out var definition))
    return;
```

`TryGetValue` пытается найти значение в словаре по ключу.

Здесь ключ — `stepId`.

Если шаг найден:

- метод вернёт `true`;
- переменная `definition` получит найденный шаг.

Если шаг не найден:

- метод вернёт `false`;
- мы выходим через `return`.

#### Что такое `out var definition`

`out` означает, что метод запишет значение в переменную.

`var definition` создаёт переменную, тип которой компилятор определит сам.

Здесь тип будет:

```csharp
ScriptStepDefinition
```

#### Что означает `!`

```csharp
!_steps.TryGetValue(...)
```

`!` — логическое НЕ.

Если `TryGetValue` вернул `false`, то `!false` станет `true`.

То есть:

```csharp
if (!_steps.TryGetValue(...))
```

читается так:

> если шаг не найден — выйти.

---

```csharp
var visibleStep = new VisibleStepViewModel(
    definition.Id,
    definition.Theme,
    ApplyVariables(definition.TextTemplate));
```

Создаём видимый шаг.

В конструктор передаём:

1. `Id`;
2. `Theme`;
3. готовый текст после подстановки переменных.

---

```csharp
ApplyVariables(definition.TextTemplate)
```

Метод заменяет:

```text
{OperatorName}
{ClientName}
```

на реальные значения.

---

```csharp
foreach (var answer in definition.Answers)
```

`foreach` — цикл перебора коллекции.

Он проходит по всем ответам шага.

Например, если ответы:

```text
Да
Нет
```

то тело цикла выполнится два раза.

---

```csharp
visibleStep.Answers.Add(
    new AnswerOptionViewModel(
        answer.Text,
        new RelayCommand(_ => SelectAnswer(visibleStep, answer))));
```

Это самая важная строка.

Она создаёт кнопку-ответ для экрана.

Разберём вложенность:

```csharp
new AnswerOptionViewModel(...)
```

Создаём ViewModel ответа.

Первый аргумент:

```csharp
answer.Text
```

Это текст кнопки.

Второй аргумент:

```csharp
new RelayCommand(_ => SelectAnswer(visibleStep, answer))
```

Это команда, которая выполнится при нажатии.

---

```csharp
_ => SelectAnswer(visibleStep, answer)
```

Это лямбда.

Символ `_` означает: параметр команды нам не нужен.

Полная форма могла бы быть такой:

```csharp
parameter => SelectAnswer(visibleStep, answer)
```

Но так как `parameter` не используется, часто пишут `_`.

То есть команда говорит:

> Когда кнопку нажмут, вызови `SelectAnswer` для текущего видимого шага и текущего ответа.

---

```csharp
VisibleSteps.Add(visibleStep);
```

Добавляем готовый шаг в коллекцию видимых шагов.

Так как `VisibleSteps` — `ObservableCollection`, WPF сам обновит экран.

## 15. Подстановка переменных `ApplyVariables`

### Коротко

Метод заменяет шаблоны в тексте на реальные значения.

Код:

```csharp
private string ApplyVariables(string template)
{
    return template
        .Replace("{OperatorName}", _operatorName)
        .Replace("{ClientName}", _clientName);
}
```

---

### Подробнее

Допустим, есть текст:

```text
Здравствуйте, меня зовут {OperatorName}. Я разговариваю с {ClientName}?
```

Оператор ввёл:

```text
Евгений
```

Клиент:

```text
Иван Иванович
```

Метод сделает:

```text
Здравствуйте, меня зовут Евгений. Я разговариваю с Иван Иванович?
```

---

```csharp
return template
    .Replace("{OperatorName}", _operatorName)
    .Replace("{ClientName}", _clientName);
```

Методы вызываются цепочкой.

Сначала:

```csharp
.Replace("{OperatorName}", _operatorName)
```

Потом на результате вызывается:

```csharp
.Replace("{ClientName}", _clientName)
```

Это называется method chaining.

Перевод: «цепочка вызовов методов».

У нас:

```xml
<RowDefinition Height="Auto"/>

```

Первая строка — высота по содержимому, шапка.  
Вторая строка — всё остальное пространство, вкладки.

```xml
<TabControl Grid.Row="1" Margin="16">
```

`TabControl` создаёт вкладки.

Каждая вкладка — это `TabItem`.

```csharp
DataContext = new MainViewModel(...);
```

## 20. Код главного окна `MainWindow.xaml.cs`

### Коротко

Код показывает стартовый диалог и создаёт `MainViewModel`.

```csharp
using System.Windows;
using OperatorScriptWpf.ViewModels;

namespace OperatorScriptWpf;

public partial class MainWindow : Window
{
    private bool _startupDialogWasShown;

    public MainWindow()
    {
        InitializeComponent();
    }

    private void MainWindow_Loaded(object sender, RoutedEventArgs e)
    {
        if (_startupDialogWasShown)
            return;

        _startupDialogWasShown = true;

        var dialog = new StartupDialog
        {
            Owner = this
        };

        bool? result = dialog.ShowDialog();

        if (result == true)
        {
            DataContext = new MainViewModel(dialog.OperatorName, dialog.ClientName);
        }
        else
        {
            DataContext = new MainViewModel("Оператор", "Клиент");
        }
    }
}
```

---

### Подробнее

```csharp
private bool _startupDialogWasShown;
```

Поле-флаг.

**Flag** — флаг, признак.

Оно нужно, чтобы стартовый диалог не открылся повторно.

Событие `Loaded` иногда может сработать повторно в некоторых сценариях. Поэтому защита полезна.

---

```csharp
if (_startupDialogWasShown)
    return;
```

Если диалог уже показывали, выходим.

---

```csharp
_startupDialogWasShown = true;
```

Помечаем, что диалог уже был показан.

---

```csharp
var dialog = new StartupDialog
{
    Owner = this
};
```

Создаём стартовое окно.

`Owner = this` означает: владельцем диалога будет текущее главное окно.

Зачем это нужно:

- диалог центрируется относительно главного окна;
- диалог ведёт себя как дочернее окно;
- пользователь не потеряет его за главным окном.

---

```csharp
bool? result = dialog.ShowDialog();
```

`ShowDialog()` показывает модальное окно.

**Modal dialog** — модальный диалог.

Это окно, которое блокирует работу с главным окном, пока его не закроют.

Возвращаемый тип — `bool?`.

Почему `bool?`, а не `bool`?

Потому что результат может быть:

- `true` — пользователь нажал сохранить;
- `false` — пользователь явно отменил;
- `null` — окно закрыли без результата.

---

```csharp
if (result == true)
```

Проверяем именно `true`.

Это безопасно для nullable bool.

---

```csharp
DataContext = new MainViewModel(dialog.OperatorName, dialog.ClientName);
```

Это критически важная строка.

`DataContext` — источник данных для binding.

Когда мы пишем в XAML:

```xml
ItemsSource="{Binding VisibleSteps}"
```

WPF ищет `VisibleSteps` в `DataContext`.

А `DataContext` здесь — это `MainViewModel`.

## 25. Минимальная проверка понимания

Ответь себе письменно.

### Вопросы

1. Чем `ScriptStepDefinition` отличается от `VisibleStepViewModel`?
2. Зачем нужен `ObservableCollection`?
3. Что делает `DataContext`?
4. Что означает `{Binding VisibleSteps}`?
5. Зачем нужен `RelayCommand`?
6. Почему при выборе ответа удаляются нижние шаги?
7. Что делает `ApplyVariables`?
8. Почему у `NextStepId` тип `string?`?
9. Что такое `init`?
10. Почему `ShowDialog()` возвращает `bool?`?

### Короткие ответы

1. `ScriptStepDefinition` — исходное описание шага, `VisibleStepViewModel` — подготовленный для показа шаг.
2. Чтобы WPF видел добавление и удаление элементов.
3. Это объект, из которого WPF берёт данные для binding.
4. Взять свойство `VisibleSteps` из `DataContext`.
5. Чтобы кнопка могла вызвать метод ViewModel.
6. Потому что старая ветка сценария больше не актуальна.
7. Заменяет placeholders на реальные имена.
8. Потому что следующего шага может не быть.
9. Свойство можно задать при создании объекта, но потом нельзя менять.
10. Потому что результат может быть `true`, `false` или `null`.

## 27. Частые ошибки

### Ошибка 1. Окно пустое

Проверь, задан ли `DataContext`:

```csharp
DataContext = new MainViewModel(...);
```

---

### Ошибка 2. Binding не работает

Проверь имя свойства.

В XAML:

```xml
{Binding VisibleSteps}
```

В C# должно быть именно:

```csharp
public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();
```

Одна лишняя буква — binding не найдёт свойство.

---

### Ошибка 3. Кнопки не появляются

Проверь, что в шаге есть `Answers`.

Финальные шаги имеют:

```csharp
Answers = []
```

Там кнопок не будет — это нормально.

---

### Ошибка 4. Нажатие на кнопку ничего не делает

Проверь, что в `AnswerOptionViewModel` есть команда:

```csharp
new RelayCommand(_ => SelectAnswer(visibleStep, answer))
```

И что XAML привязан к ней:

```xml
Command="{Binding SelectCommand}"
```

---

### Ошибка 5. Следующий шаг не появляется

Проверь, что `NextStepId` совпадает с `Id` существующего шага.

Например:

```csharp
NextStepId = "ready"
```

и где-то должен быть шаг:

```csharp
Id = "ready"
```

### Коротко

Предыдущий учебник хорошо разбирал рабочий код, но ему не хватало общей картины: как WPF устроен внутри, как слои взаимодействуют между собой и как развивать проект не «водопадом», а маленькими рабочими шагами.

### Подробнее

В реальной разработке junior-разработчик часто ошибается не потому, что не знает синтаксис C#, а потому что не видит связей:

- где заканчивается окно и начинается логика;
- почему `Button` не должен напрямую знать всё о сценарии;
- зачем нужен `DataContext`;
- зачем нужна `ViewModel`;
- почему коллекция должна быть `ObservableCollection`, а не обычный `List`;
- почему сначала лучше сделать маленький вертикальный срез, а не проектировать всю систему на месяц вперёд.

Поэтому дальше учебник расширен по принципу:

```text
сначала понять общую идею
потом сделать маленький рабочий кусок
потом выделить слой
потом улучшить архитектуру
потом добавить следующую возможность
```

Это и есть практичный agile-подход: короткие итерации, рабочий результат после каждого шага, возможность проверить себя без интернета и без наставника рядом.

# 1. Главная идея WPF-приложения

## Коротко

WPF-приложение — это desktop-программа для Windows, где внешний вид описывается в XAML, а поведение пишется на C#.

В простом виде:

```text
XAML показывает интерфейс
C# хранит данные и выполняет действия
Binding связывает XAML и C#
```

## Подробнее

WPF расшифровывается как **Windows Presentation Foundation** — «основа представления Windows». Это технология для создания оконных приложений под Windows.

В WPF обычно есть две половины:

```text
View        — внешний вид, XAML
Logic       — поведение, C#
```

Но в нормальном проекте C# тоже делится на части:

```text
View        — окно и разметка
ViewModel   — состояние экрана и команды
Model       — данные предметной области
Service     — техническая или прикладная логика
```

Для нашего проекта это выглядит так:

```text
MainWindow.xaml
    показывает вкладки, карточки шагов и кнопки ответов

MainViewModel.cs
    хранит видимые шаги и реагирует на выбор ответа

ScriptStepDefinition.cs
    описывает один шаг сценария

ScriptAnswer.cs
    описывает один ответ оператора/клиента

ScriptService.cs
    создаёт сценарий и отдаёт его приложению
```

Запомни важное правило: **WPF сам не является архитектурой**. WPF даёт инструменты: окна, элементы управления, привязки, стили, шаблоны. Архитектуру выбирает разработчик.

# 2. Как запускается WPF-приложение

## Коротко

При запуске WPF сначала создаёт объект приложения `App`, затем открывает стартовое окно, указанное в `StartupUri`.

## Подробнее

В проекте есть файл:

```xml
<Application x:Class="OperatorScriptWpf.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             StartupUri="MainWindow.xaml">
</Application>
```

Ключевая строка:

```xml
StartupUri="MainWindow.xaml"
```

Она означает: «при запуске приложения открой `MainWindow.xaml`».

УПРОЩЕНИЕ: можно представить, что WPF делает примерно так:

```csharp
var app = new App();
var window = new MainWindow();
window.Show();
app.Run();
```

Так писать вручную в этом проекте не нужно. Это объяснительная модель, чтобы понять порядок событий.

Реальный порядок для нашего приложения:

```text
1. Пользователь запускает exe или dotnet run.
2. .NET стартует приложение.
3. WPF читает App.xaml.
4. WPF видит StartupUri="MainWindow.xaml".
5. WPF создаёт MainWindow.
6. MainWindow вызывает InitializeComponent().
7. WPF строит визуальное дерево из XAML.
8. Окно загружается.
9. Срабатывает событие Loaded.
10. Мы показываем StartupDialog.
11. Пользователь вводит имя оператора и клиента.
12. Мы создаём MainViewModel.
13. Устанавливаем DataContext.
14. Binding показывает данные на экране.
```

# 3. Что такое визуальное дерево WPF

## Коротко

WPF-окно состоит из дерева элементов: `Window` содержит `Grid`, `Grid` содержит `TabControl`, `TabControl` содержит `TabItem`, а внутри уже карточки и кнопки.

## Подробнее

Пример из нашего проекта:

```text
Window
└── Grid
    ├── Border
    │   └── TextBlock
    └── TabControl
        ├── TabItem: Представление
        │   └── ScrollViewer
        │       └── ItemsControl
        │           └── карточки шагов
        ├── TabItem: Дозвон, Успешно
        ├── TabItem: Дозвон, Отказ
        ├── TabItem: Недозвон
        ├── TabItem: Перезвон
        └── TabItem: Ошибка телефонии
```

Это называется **visual tree** — визуальное дерево.

Почему это важно:

- свойства могут наследоваться вниз по дереву;
- `DataContext` часто наследуется дочерними элементами;
- `Binding` ищет данные в текущем контексте;
- события могут подниматься или спускаться по дереву;
- `DataTemplate` создаёт новые элементы внутри дерева автоматически.

Когда ты видишь XAML, старайся читать его как вложенную структуру, а не как плоский текст.

# 4. Что такое DataContext

## Коротко

`DataContext` — это объект, из которого WPF берёт данные для `Binding`.

Если написать:

```xml
<TextBlock Text="{Binding Title}" />
```

то WPF ищет свойство `Title` в текущем `DataContext`.

## Подробнее

В нашем проекте в `MainWindow.xaml.cs` есть строка:

```csharp
DataContext = new MainViewModel(dialog.OperatorName, dialog.ClientName);
```

Она означает:

```text
Для этого окна главным объектом данных будет MainViewModel.
```

После этого XAML может писать:

```xml
<ItemsControl ItemsSource="{Binding VisibleSteps}">
```

WPF читает это так:

```text
Возьми текущий DataContext.
Это MainViewModel.
Найди у него свойство VisibleSteps.
Покажи элементы из этой коллекции.
```

Если `DataContext` не установлен, `Binding` часто просто ничего не покажет. Это одна из самых частых ошибок новичка.

# 5. Binding: как XAML связан с C#

## Коротко

`Binding` — это привязка свойства элемента интерфейса к свойству C#-объекта.

Пример:

```xml
<TextBlock Text="{Binding Text}" />
```

означает: «текст этого `TextBlock` возьми из свойства `Text` текущего объекта».

## Подробнее

Внутри `ItemsControl` текущий объект меняется.

Снаружи:

```xml
<ItemsControl ItemsSource="{Binding VisibleSteps}">
```

`VisibleSteps` — это коллекция шагов из `MainViewModel`.

А внутри шаблона:

```xml
<DataTemplate>
    <TextBlock Text="{Binding Text}" />
</DataTemplate>
```

Текущий `DataContext` уже не `MainViewModel`, а один элемент из `VisibleSteps`, то есть `VisibleStepViewModel`.

Поэтому `Text="{Binding Text}"` означает:

```text
Возьми свойство Text у конкретного видимого шага.
```

Это важный момент. В WPF `DataContext` может меняться на разных уровнях дерева.

# 6. ItemsControl: как показать список элементов

## Коротко

`ItemsControl` показывает коллекцию. Каждый элемент коллекции отрисовывается по шаблону `DataTemplate`.

## Подробнее

В нашем проекте:

```xml
<ItemsControl ItemsSource="{Binding VisibleSteps}">
    <ItemsControl.ItemTemplate>
        <DataTemplate>
            ...
        </DataTemplate>
    </ItemsControl.ItemTemplate>
</ItemsControl>
```

Это читается так:

```text
Возьми коллекцию VisibleSteps.
Для каждого элемента создай визуальный блок по DataTemplate.
```

Если в `VisibleSteps` три шага, WPF создаст три карточки.

Если ViewModel добавит четвёртый шаг:

```csharp
VisibleSteps.Add(visibleStep);
```

WPF автоматически покажет четвёртую карточку, потому что коллекция `ObservableCollection` умеет уведомлять интерфейс об изменениях.

# 7. ObservableCollection: почему не обычный List

## Коротко

`ObservableCollection<T>` уведомляет WPF, когда элемент добавили или удалили.

`List<T>` этого не делает.

## Подробнее

Допустим, у нас есть обычный список:

```csharp
public List<VisibleStepViewModel> VisibleSteps { get; } = new();
```

Если после запуска сделать:

```csharp
VisibleSteps.Add(newStep);
```

данные в памяти изменятся, но WPF может не обновить экран.

Поэтому для интерфейса используется:

```csharp
public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();
```

Слово `Observable` значит «наблюдаемый». Коллекция сообщает подписчикам:

```text
добавлен элемент
удалён элемент
коллекция очищена
```

WPF подписывается на эти сообщения и обновляет экран.

# 8. INotifyPropertyChanged: когда меняется свойство

## Коротко

`INotifyPropertyChanged` нужен, чтобы WPF узнавал об изменении свойства объекта.

## Подробнее

`ObservableCollection` сообщает об изменениях коллекции. Но если меняется отдельное свойство объекта, нужен другой механизм.

Например:

```csharp
private string _title = "Старый текст";

public string Title
{
    get => _title;
    set
    {
        _title = value;
        OnPropertyChanged();
    }
}
```

Когда вызывается `OnPropertyChanged()`, WPF получает сигнал:

```text
Свойство Title изменилось. Нужно перечитать значение и обновить интерфейс.
```

В нашем проекте `MainViewModel` наследуется от `ObservableObject`, где уже реализована эта механика.

Даже если сейчас у нас мало изменяемых свойств, базовый класс полезен: проект будет расти, и свойства вроде `SelectedTab`, `StatusText`, `OperatorName`, `IsBusy` появятся почти неизбежно.

# 9. Команды вместо Click-обработчиков

## Коротко

Для учебного старта можно использовать `Click`, но для WPF/MVVM лучше использовать `Command`.

## Подробнее

Простой подход:

```xml
<Button Content="Сохранить" Click="SaveButton_Click" />
```

И в code-behind:

```csharp
private void SaveButton_Click(object sender, RoutedEventArgs e)
{
    // логика
}
```

Это нормально для маленького диалога `StartupDialog`, потому что там логика минимальная: забрать текст и закрыть окно.

Но для сценария ответов лучше так:

```xml
<Button Content="{Binding Text}"
        Command="{Binding SelectCommand}" />
```

Кнопка не знает, что именно произойдёт. Она просто вызывает команду.

Команда лежит в `AnswerOptionViewModel`:

```csharp
public ICommand SelectCommand { get; }
```

А создаётся она в `MainViewModel`:

```csharp
new RelayCommand(_ => SelectAnswer(visibleStep, answer))
```

Это означает:

```text
Когда нажмут этот ответ, вызови SelectAnswer для текущего шага и текущего ответа.
```

Так UI остаётся простым, а логика находится во ViewModel.

# 10. Слои проекта: кто за что отвечает

## Коротко

В проекте должны быть слои. Не нужно складывать всё в `MainWindow.xaml.cs`.

## Подробнее

Минимальные слои для нашего проекта:

```text
View
    XAML-окна. Показывают интерфейс.

ViewModel
    Хранит состояние экрана и команды.

Model
    Описывает данные: шаг, ответ, результат.

Service
    Загружает сценарий, сохраняет данные, работает с файлами или API.
```

Для текущего приложения:

```text
View:
    MainWindow.xaml
    StartupDialog.xaml

ViewModel:
    MainViewModel
    VisibleStepViewModel
    AnswerOptionViewModel

Model:
    ScriptStepDefinition
    ScriptAnswer

Service:
    ScriptService
```

Правило:

```text
View может знать про ViewModel.
ViewModel может знать про Model и Service.
Service может знать про Model.
Model не должна знать про View, ViewModel и Service.
```

Схема зависимостей:

```text
View  --->  ViewModel  --->  Service  --->  Model
              |
              +--------->  Model
```

Плохая схема:

```text
ScriptService напрямую меняет TextBlock
Model открывает MessageBox
ScriptAnswer вызывает MainWindow
ViewModel лезет в конкретные TextBox по имени
```

Почему это плохо:

- трудно тестировать;
- трудно менять интерфейс;
- трудно переносить логику в другой UI;
- код быстро становится запутанным.

# 11. Agile-подход для этого проекта

## Коротко

Не надо сначала писать весь проект целиком. Делаем маленькие рабочие версии.

## Подробнее

Водопадный подход выглядел бы так:

```text
1. Сначала полностью спроектировать все классы.
2. Потом полностью написать все модели.
3. Потом полностью написать весь интерфейс.
4. Потом подключить логику.
5. Потом впервые запустить.
```

Это опасно для новичка: можно неделю писать код и не понимать, где ошибка.

Agile-подход:

```text
1. Сделали маленький рабочий кусок.
2. Запустили.
3. Проверили.
4. Поняли, что работает.
5. Улучшили структуру.
6. Добавили следующую возможность.
```

Для нашего WPF-проекта это будет так:

```text
Итерация 0.1 — пустое WPF-окно запускается.
Итерация 0.2 — добавили вкладки.
Итерация 0.3 — показали один статический шаг.
Итерация 0.4 — показали список шагов через ItemsControl.
Итерация 0.5 — добавили кнопку ответа.
Итерация 0.6 — сделали добавление следующего шага.
Итерация 0.7 — сделали удаление лишних шагов.
Итерация 0.8 — добавили стартовое окно ввода имени.
Итерация 0.9 — вынесли сценарий в Service.
Итерация 1.0 — получили рабочий MVP.
```

MVP — **Minimum Viable Product**, минимально жизнеспособный продукт. В нашем случае это не идеальная программа, а версия, которой уже можно пользоваться для демонстрации логики сценария.

# 12. Вертикальный срез: что это значит

## Коротко

Вертикальный срез — это маленькая функция, проходящая через все слои: от кнопки в UI до логики в коде и обратно на экран.

## Подробнее

Например, функция «нажать ответ и показать следующий шаг» затрагивает:

```text
XAML:
    Button

ViewModel ответа:
    SelectCommand

MainViewModel:
    SelectAnswer

Model:
    ScriptAnswer.NextStepId

Service:
    словарь всех шагов

ObservableCollection:
    VisibleSteps.Add(...)

WPF Binding:
    обновление экрана
```

Это маленькая функция, но она проходит через все важные части системы. Поэтому она ценнее, чем десять заранее написанных классов, которые ещё ни разу не запускались.

В обучении это особенно важно: каждый вертикальный срез даёт понимание, а не просто набор файлов.

# 16. Итерация 0.4 — список шагов

## Коротко

Переходим от одного статического блока к коллекции `VisibleSteps`.

## Подробнее

Нам нужна коллекция:

```csharp
public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();
```

И XAML:

```xml
<ItemsControl ItemsSource="{Binding VisibleSteps}">
    <ItemsControl.ItemTemplate>
        <DataTemplate>
            <TextBlock Text="{Binding Text}" />
        </DataTemplate>
    </ItemsControl.ItemTemplate>
</ItemsControl>
```

Теперь экран не «зашит» на один текст. Он показывает столько шагов, сколько есть в коллекции.

Это первый серьёзный переход от ручного UI к data-driven UI.

**Data-driven UI** — интерфейс, управляемый данными. Перевод: «интерфейс, который строится на основе данных».

# 17. Итерация 0.5 — ответы как команды

## Коротко

У каждого шага появляется список ответов, а у каждого ответа — команда.

## Подробнее

Модель ответа:

```csharp
public sealed class ScriptAnswer
{
    public string Text { get; init; } = "";
    public string? NextStepId { get; init; }
}
```

ViewModel ответа:

```csharp
public sealed class AnswerOptionViewModel
{
    public AnswerOptionViewModel(string text, ICommand selectCommand)
    {
        Text = text;
        SelectCommand = selectCommand;
    }

    public string Text { get; }
    public ICommand SelectCommand { get; }
}
```

Почему есть две сущности:

```text
ScriptAnswer
    чистое описание ответа из сценария

AnswerOptionViewModel
    ответ, подготовленный для отображения в WPF, с командой для кнопки
```

Это важное разделение. Модель не должна знать про `ICommand`, потому что `ICommand` — это WPF/UI-механика.

# 18. Итерация 0.6 — выбор ответа и следующий шаг

## Коротко

При нажатии ответа мы берём `NextStepId` и добавляем следующий шаг в `VisibleSteps`.

## Подробнее

Ключевой код:

```csharp
if (string.IsNullOrWhiteSpace(answer.NextStepId))
    return;

AddStep(answer.NextStepId);
```

Что здесь происходит:

```text
1. Если у ответа нет следующего шага — ничего не добавляем.
2. Если следующий шаг указан — вызываем AddStep.
```

Метод `AddStep` ищет описание шага:

```csharp
if (!_steps.TryGetValue(stepId, out var definition))
    return;
```

Словарь `_steps` нужен, чтобы быстро найти шаг по `Id`.

Почему не просто список:

```text
List — нужно искать перебором.
Dictionary — можно обратиться по ключу.
```

Для сценария с переходами `Dictionary` подходит лучше.

# 19. Итерация 0.7 — скрытие лишних шагов

## Коротко

Если пользователь вернулся к раннему шагу и выбрал другой ответ, все шаги после него нужно удалить.

## Подробнее

Это требование есть в исходной задаче: если был открыт более поздний шаг, а потом выбран пункт из предыдущего шага, лишние открытые шаги нужно скрыть.

Код:

```csharp
int currentIndex = VisibleSteps.IndexOf(currentStep);

if (currentIndex < 0)
    return;

while (VisibleSteps.Count > currentIndex + 1)
{
    VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
}
```

Разберём по шагам.

```csharp
int currentIndex = VisibleSteps.IndexOf(currentStep);
```

Мы узнаём позицию текущего шага в списке видимых шагов.

Пример:

```text
0: start
1: ready
2: offer
3: price
```

Если пользователь нажал ответ в шаге `ready`, индекс будет `1`.

Дальше:

```csharp
while (VisibleSteps.Count > currentIndex + 1)
```

Пока количество видимых шагов больше, чем должно остаться, удаляем последний.

Если `currentIndex = 1`, должны остаться шаги с индексами `0` и `1`.

Удаляем:

```text
price
offer
```

После этого добавляем новый следующий шаг.

Это не просто техническая деталь. Это бизнес-логика интерактивного сценария.

# 20. Итерация 0.8 — стартовый диалог

## Коротко

Перед началом работы оператор вводит своё имя и имя клиента. Эти значения подставляются в сценарий.

## Подробнее

В WPF модальное окно можно показать так:

```csharp
var dialog = new StartupDialog
{
    Owner = this
};

```

`ShowDialog()` открывает окно модально. Это значит:

```text

```

Результат имеет тип `bool?`.

Почему не просто `bool`?

Потому что возможны три состояния:

```text
false  — пользователь отменил действие
null   — окно закрыто без явного результата
```

```csharp
else

}
```

Если пользователь нажал «Сохранить», берём введённые данные. Если закрыл окно иначе — используем значения по умолчанию.

# 21. Итерация 0.9 — сервис сценария

## Коротко

```text
    знает, что делать при выборе ответа
```

# 23. Как думать при добавлении новой функции

## Коротко

Перед кодом задай себе четыре вопроса: что меняется в данных, что меняется на экране, какая команда нужна, какой слой отвечает.

## Подробнее

Пример: нужно добавить кнопку «Начать заново».

Вопросы:

```text
1. Что меняется в данных?
   Нужно очистить VisibleSteps и добавить start.

2. Что меняется на экране?
   Снова должен остаться только первый шаг.

3. Какая команда нужна?
   RestartCommand.

4. Где должна быть логика?
   В MainViewModel, потому что это состояние экрана сценария.
```

Примерный код:

```csharp
public ICommand RestartCommand { get; }

public MainViewModel(string operatorName, string clientName)
{
    ...
    RestartCommand = new RelayCommand(_ => Restart());
    AddStep("start");
}

private void Restart()
{
    VisibleSteps.Clear();
    AddStep("start");
}
```

В XAML:

```xml
<Button Content="Начать заново"
        Command="{Binding RestartCommand}" />
```

Так ты добавляешь функцию вертикальным срезом:

```text
кнопка → команда → метод → коллекция → обновление UI
```

```csharp
public sealed class RelayCommand : ICommand
{
    private readonly Action<object?> _execute;
    private readonly Predicate<object?>? _canExecute;

    public RelayCommand(Action<object?> execute, Predicate<object?>? canExecute = null)
    {
        _execute = execute ?? throw new ArgumentNullException(nameof(execute));
        _canExecute = canExecute;
    }

    public bool CanExecute(object? parameter)
    {
        return _canExecute?.Invoke(parameter) ?? true;
    }

    public void Execute(object? parameter)
    {
        _execute(parameter);
    }

    public event EventHandler? CanExecuteChanged;

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

### `ViewModels/MainViewModel.cs`

```csharp
using System.Collections.ObjectModel;
using OperatorScriptWpf.Models;
using OperatorScriptWpf.Services;

namespace OperatorScriptWpf.ViewModels;

public sealed class MainViewModel : ObservableObject
{
    private readonly IReadOnlyDictionary<string, ScriptStepDefinition> _steps;
    private readonly string _operatorName;
    private readonly string _clientName;

    public MainViewModel(string operatorName, string clientName)
    {
        _operatorName = string.IsNullOrWhiteSpace(operatorName) ? "Оператор" : operatorName.Trim();
        _clientName = string.IsNullOrWhiteSpace(clientName) ? "Клиент" : clientName.Trim();

        var scriptService = new ScriptService();
        _steps = scriptService.LoadPresentationScript();

        AddStep("start");
    }

    public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();

    private void SelectAnswer(VisibleStepViewModel currentStep, ScriptAnswer answer)
    {
        int currentIndex = VisibleSteps.IndexOf(currentStep);

        if (currentIndex < 0)
            return;

        while (VisibleSteps.Count > currentIndex + 1)
        {
            VisibleSteps.RemoveAt(VisibleSteps.Count - 1);
        }

        if (string.IsNullOrWhiteSpace(answer.NextStepId))
            return;

        AddStep(answer.NextStepId);
    }

    private void AddStep(string stepId)
    {
        if (!_steps.TryGetValue(stepId, out var definition))
            return;

        var visibleStep = new VisibleStepViewModel(
            definition.Id,
            definition.Theme,
            ApplyVariables(definition.TextTemplate));

        foreach (var answer in definition.Answers)
        {
            visibleStep.Answers.Add(
                new AnswerOptionViewModel(
                    answer.Text,
                    new RelayCommand(_ => SelectAnswer(visibleStep, answer))));
        }

        VisibleSteps.Add(visibleStep);
    }

    private string ApplyVariables(string template)
    {
        return template
            .Replace("{OperatorName}", _operatorName)
            .Replace("{ClientName}", _clientName);
    }
```

# 27. Как читать XAML начинающему

## Коротко

Читай XAML как дерево: внешний контейнер → вложенные элементы → свойства → привязки.

## Подробнее

Пример:

```xml
<Border Margin="0,0,0,14"
        Padding="16"
        Background="White">
    <StackPanel>
        <TextBlock Text="{Binding Text}" />
    </StackPanel>
</Border>
```

Как читать:

```text
Border — рамка/контейнер карточки.
Margin — внешний отступ.
Padding — внутренний отступ.
Background — фон.
StackPanel — складывает элементы внутри по вертикали.
TextBlock — показывает текст.
Text="{Binding Text}" — текст берётся из свойства Text текущего объекта.
```

Если в XAML видишь `{Binding ...}`, всегда спрашивай:

```text
Какой здесь DataContext?
У какого объекта ищется это свойство?
```

Это ключевой навык WPF.

# 28. Как читать C# начинающему

## Коротко

Читай C# по ответственности класса: что класс хранит, что получает в конструкторе, какие методы вызываются снаружи, какие методы скрыты внутри.

## Подробнее

Пример:

```csharp
public sealed class MainViewModel : ObservableObject
```

Разбор:

```text
public
    класс доступен из других частей проекта

sealed
    от этого класса нельзя наследоваться

class
    объявляется класс

MainViewModel
    имя класса

: ObservableObject
    класс наследуется от ObservableObject
```

Дальше смотри поля:

```csharp
private readonly IReadOnlyDictionary<string, ScriptStepDefinition> _steps;
```

Разбор:

```text
private
    поле доступно только внутри класса

readonly
    поле можно присвоить только при объявлении или в конструкторе

IReadOnlyDictionary<string, ScriptStepDefinition>
    словарь только для чтения: ключ string, значение ScriptStepDefinition

_steps
    имя поля; подчёркивание показывает, что это приватное поле
```

Потом конструктор:

```csharp
public MainViewModel(string operatorName, string clientName)
```

Конструктор отвечает за начальную подготовку объекта.

Потом публичные свойства:

```csharp
public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();
```

Это свойство видно XAML через `Binding`.

Потом приватные методы:

```csharp
private void SelectAnswer(...)
private void AddStep(...)
private string ApplyVariables(...)
```

Они скрыты внутри, потому что являются внутренними деталями работы ViewModel.

# 30. Когда выделять новый слой или класс

## Коротко

Новый класс нужен не «потому что красиво», а когда у кода появляется отдельная ответственность.

## Подробнее

Признаки, что пора выделять класс:

```text
1. Метод стал длиннее 30-50 строк.
2. В классе смешались разные причины изменения.
3. Код хочется протестировать отдельно.
4. Код используется в двух местах.
5. Название метода стало слишком общим: Process, Handle, DoWork.
6. В комментариях приходится объяснять большой алгоритм.
```

Пример:

Сначала логика переходов находится в `MainViewModel`. Это нормально для MVP.

Потом проект растёт:

```text
появились условия переходов
появились обязательные вопросы
появились результаты звонка
появилось сохранение истории
```

Тогда можно выделить:

```text
ScriptNavigator
    отвечает только за переходы между шагами

ScriptSession
    хранит состояние одного прохождения сценария

ScriptResult
    хранит итог звонка
```

Это agile-рефакторинг: мы не проектируем всё заранее, а улучшаем структуру, когда появилась реальная необходимость.

# 35. НУЖНО ДОУЧИТЬ

## 1. Dependency Properties

**Dependency Property** — свойство зависимостей. Это специальная система свойств WPF, на которой работают стили, привязки, анимации и наследование значений.

Для текущего проекта глубокое знание пока не требуется, но позже оно понадобится.

## 2. Routed Events

**Routed Events** — маршрутизируемые события. Это события, которые могут идти по дереву элементов вверх или вниз.

Например, клик по кнопке может быть обработан не только самой кнопкой, но и родительским контейнером.

## 3. Styles

**Style** — стиль. Позволяет описать внешний вид элементов один раз и применять ко многим элементам.

## 4. Templates

**Template** — шаблон. Позволяет полностью менять визуальное представление элемента управления.

## 5. Resources

**Resources** — ресурсы. Это словари объектов, стилей, цветов, шаблонов, которые можно переиспользовать.

## 6. Validation

**Validation** — валидация. Проверка корректности данных, введённых пользователем.

## 7. Async/await

**Async/await** — асинхронность. Понадобится, когда WPF будет обращаться к API, файлам, базе или устройствам, не замораживая интерфейс.

# 36. Финальный ориентир

## Коротко

Ты не просто копируешь WPF-код. Ты строишь маленькую систему слоями и развиваешь её итерациями.

## Подробнее

Правильная мысль для этого проекта:

```text
Сначала рабочее окно.
Потом рабочие вкладки.
Потом один шаг.
Потом список шагов.
Потом ответы.
Потом переходы.
Потом подстановка переменных.
Потом сервис.
Потом улучшение архитектуры.
```

Это путь от простого к сложному. Он защищает от хаоса и даёт понимание.

Если в какой-то момент код становится непонятным, не добавляй новую функцию. Остановись и ответь:

```text
Какой слой я сейчас меняю?
Какая ответственность у этого класса?
Какие данные входят?
Какие данные выходят?
Как я проверю, что это работает?
```

Так думает разработчик, который растёт от копирования к самостоятельному проектированию.

# РЕДАКТОРСКАЯ ДОРАБОТКА: КАК ПОЛЬЗОВАТЬСЯ УЧЕБНИКОМ ОФФЛАЙН

## Коротко

Этот раздел нужно прочитать **перед уроком 1**. Он задаёт правильный режим работы: не копировать весь код сразу, а идти маленькими рабочими слоями. Каждый урок должен заканчиваться запущенной программой. Если программа не запускается, следующий урок начинать нельзя.

Главный цикл обучения:

```text
1. Прочитал цель урока.
2. Прочитал короткую теорию.
3. Написал код руками.
4. Запустил приложение.
5. Проверил, что изменилось.
6. Специально сломал одну вещь.
7. Посмотрел ошибку.
8. Исправил.
9. Ответил на вопросы самопроверки.
10. Только после этого перешёл дальше.
```

## Подробно

Оффлайн-обучение опасно тем, что рядом нет преподавателя и нельзя быстро открыть Stack Overflow, Microsoft Learn или форум. Поэтому учебник должен заменить три вещи:

1. **маршрут** — что делать сейчас и что делать потом;
2. **справочник** — что означает каждая новая конструкция;
3. **наставника** — почему мы пишем именно так, а не иначе.

Работай с каждым уроком по правилу: **один урок — один проверяемый результат**.

Нельзя делать так:

```text
Прочитал 50 страниц → скопировал финальный проект → не понял, почему работает.
```

Нужно делать так:

```text
Урок 1: окно запустилось.
Урок 2: появились вкладки.
Урок 3: появился диалог.
Урок 4: введённые имена попали в главное окно.
Урок 5: появился первый шаг сценария.
Урок 6: шаги стали моделями.
Урок 7: сценарий ушёл в сервис.
Урок 8: видимые шаги стали коллекцией.
Урок 9: состояние переехало во ViewModel.
Урок 10: кнопки стали командами.
Урок 11: View связалась с ViewModel через Binding.
Урок 12: проект собран слоями.
```

Такой подход ближе к реальной разработке: сначала рабочий результат, потом улучшение архитектуры. Это и есть практический agile-подход: короткая итерация, проверка, исправление, следующий слой.

## Как вести рабочую тетрадь оффлайн

Для каждого урока заведи короткую запись:

```text
Урок: 4. Подключаем стартовый диалог
Дата:
Что должно заработать:
Что я изменил:
Какая ошибка возникла:
Как я её исправил:
Что я понял:
Что осталось непонятно:
```

Это важно. Если через три дня проект перестанет собираться, такая запись поможет восстановить, на каком слое возникла проблема.

## Как проверять себя после каждого урока

После каждого урока ответь письменно на четыре вопроса:

1. **Что добавил этот слой?**
2. **Какой файл стал главным в этом уроке?**
3. **Какая новая WPF-конструкция появилась?**
4. **Как проверить, что слой работает?**

Пример для урока с вкладками:

```text
Что добавил слой?
— В главное окно добавлен TabControl с вкладками.

Главный файл?
— MainWindow.xaml.

Новая конструкция?
— TabControl и TabItem.

Как проверить?
— Запустить dotnet run и щёлкнуть по вкладкам.
```

## Правило «сначала сломай, потом пойми»

Когда код заработал, полезно намеренно сломать одну вещь:

- удалить закрывающий тег в XAML;
- поменять имя свойства в Binding;
- убрать `DataContext`;
- заменить `ObservableCollection` на `List`;
- удалить `OnPropertyChanged`;
- изменить `NextStepId` на несуществующий идентификатор.

Потом посмотри, какая ошибка появилась, и верни код обратно. Это не вредительство, а обучение диагностике. В реальной WPF-разработке половина времени уходит не на написание кода, а на понимание: почему окно пустое, почему Binding не сработал, почему команда не вызвалась.

# РЕДАКТОРСКИЙ СТАНДАРТ КАЖДОГО УРОКА

## Коротко

Каждый урок в этом учебнике нужно читать по одинаковой схеме. Если в каком-то уроке схема кажется неполной, используй этот раздел как чек-лист.

Шаблон урока:

```text
1. Цель урока.
2. Что получится в конце.
3. Новые термины.
4. Теория по Microsoft Learn простыми словами.
5. Практика.
6. Код.
7. Построчный разбор.
8. Проверка запуска.
9. Мини-задания.
10. Частые ошибки.
11. Итог урока.
```

## Подробно

### 1. Цель урока

Цель отвечает на вопрос: **зачем этот слой нужен приложению?**

Пример:

```text
Цель: добавить стартовый диалог, чтобы оператор мог ввести своё имя и имя клиента.
```

Это лучше, чем сухое:

```text
Сегодня изучаем Window.ShowDialog().
```

Потому что в реальном проекте мы изучаем технологию через задачу.

### 2. Что получится в конце

Перед практикой должно быть понятно, какой видимый результат появится.

Пример:

```text
После урока при запуске программы появится отдельное окно с двумя полями и кнопкой «Сохранить».
```

Если в конце урока нет запускаемого результата, это не рабочий слой, а теоретический блок. Теоретические блоки допустимы, но они не должны заменять практические слои.

### 3. Новые термины

Каждый новый термин должен быть переведён и объяснён.

Пример:

- `Window` — окно;
- `Dialog` — диалоговое окно;
- `ShowDialog()` — показать окно модально, то есть заблокировать работу с основным окном, пока диалог не закрыт;
- `DataContext` — контекст данных, источник свойств для Binding;
- `Binding` — привязка данных между XAML и объектом C#.

### 4. Теория по Microsoft Learn простыми словами

Microsoft Learn обычно даёт точное, но не всегда учебное объяснение. В учебнике нужно сначала дать смысл:

```text
WPF строит интерфейс из объектов. XAML — это удобный способ описать эти объекты декларативно.
```

И только потом техническую деталь:

```text
При сборке XAML преобразуется в скомпилированный ресурс BAML, а InitializeComponent() загружает его и создаёт дерево объектов.
```

### 5. Практика

Практика должна быть маленькой. Хороший размер практики для одного слоя:

- 1–3 файла;
- 20–120 строк кода;
- один видимый результат.

Если в уроке появляется сразу 8 файлов, начинающий перестаёт понимать причинно-следственные связи.

### 6. Код

Код должен быть полным для текущего слоя. Не фрагмент без контекста, а файл целиком, если это возможно.

Хорошо:

```csharp
public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();
    }
}
```

Плохо:

```csharp
InitializeComponent();
```

Потому что начинающий не понимает, где это писать.

### 7. Построчный разбор

Построчный разбор нужен не для каждой очевидной строки, а для каждой новой конструкции.

Например, если впервые встречается `partial`, надо объяснить:

```text
partial означает, что один класс разделён на несколько файлов. В WPF это нужно потому, что часть класса MainWindow создаётся из XAML автоматически, а часть пишем мы в MainWindow.xaml.cs.
```

### 8. Проверка запуска

Каждый урок должен иметь конкретную команду проверки:

```bat
dotnet build
dotnet run
```

И ожидаемый результат:

```text
Ожидаемый результат: открывается окно с шестью вкладками.
```

### 9. Мини-задания

Мини-задания нужны, чтобы уйти от копипаста.

Пример:

```text
1. Переименуй вкладку «Перезвон» в «Запланировать перезвон».
2. Добавь временный TextBlock во вкладку «Недозвон».
3. Запусти приложение и проверь, что вкладка переключается.
```

### 10. Частые ошибки

У каждого слоя есть типичные ошибки. Их лучше указывать сразу.

Пример для Binding:

```text
Ошибка: на экране пусто.
Причина: DataContext не назначен.
Проверка: поставь точку останова в конструкторе MainWindow и убедись, что DataContext = viewModel.
```

### 11. Итог урока

Итог должен связывать слой с общей архитектурой.

Пример:

```text
Теперь View умеет показывать данные из ViewModel. Это важный шаг к MVVM: окно больше не хранит сценарий внутри себя.
```

# УСИЛЕННАЯ КАРТА КУРСА: ОТ ЗАДАЧИ К СЛОЯМ

## Коротко

Техническое задание требует интерактивный скрипт оператора: вкладки, стартовое окно, подстановку имени оператора и клиента, показ следующего шага по ответу и скрытие лишних шагов при выборе более раннего ответа. В WPF мы реализуем это не как одну большую форму, а как последовательность рабочих слоёв.

## Подробно

В терминах проекта задача раскладывается так:

```text
ТЗ
↓
Окно приложения
↓
Вкладки
↓
Стартовый диалог
↓
Модели сценария
↓
Сервис сценария
↓
ViewModel состояния экрана
↓
Команды ответов
↓
Binding во View
↓
Рабочее WPF-приложение
```

Каждый слой отвечает на один вопрос.

| Слой | Вопрос | Главный WPF/C# механизм |
|---|---|---|
| Окно | Где живёт интерфейс? | `Window` |
| Вкладки | Как переключать разделы? | `TabControl`, `TabItem` |
| Диалог | Как получить имя оператора и клиента? | `ShowDialog()` |
| Модели | Как описать шаг и ответ данными? | C# classes |
| Сервис | Где хранить сценарий? | `ScriptService` |
| ViewModel | Где хранить состояние экрана? | MVVM, `INotifyPropertyChanged` |
| Коллекция | Как показывать несколько открытых шагов? | `ObservableCollection` |
| Команды | Как кнопка вызывает логику без `Click`? | `ICommand`, `RelayCommand` |
| Binding | Как XAML видит C#-свойства? | `{Binding ...}` |
| Стили | Как привести внешний вид к макету? | `Style`, `ResourceDictionary` |

Главная мысль: **слой не добавляется ради архитектуры**. Слой добавляется тогда, когда он решает конкретную проблему.

Плохо:

```text
Сначала создадим 20 папок, потому что так делают взрослые проекты.
```

Лучше:

```text
Сценарий стал большим → выносим его в ScriptService.
Код окна стал хранить состояние → выносим состояние в MainViewModel.
Кнопки начали содержать обработчики → переводим действия в ICommand.
```

# СХЕМЫ ВЗАИМОДЕЙСТВИЯ WPF ДЛЯ ЭТОГО ПРОЕКТА

## Коротко

WPF проще понимать не как набор файлов, а как цепочку взаимодействий. Ниже — схемы, которые нужно держать в голове при чтении уроков.

## Подробно

### Схема 1. Как запускается приложение

```text
dotnet run
  ↓
запускается .NET-приложение
  ↓
создаётся объект App
  ↓
WPF читает App.xaml
  ↓
по StartupUri или вручную создаётся MainWindow
  ↓
MainWindow вызывает InitializeComponent()
  ↓
XAML загружается и превращается в дерево объектов
  ↓
окно показывается пользователю
```

Что важно понять: `MainWindow.xaml` сам по себе не является исполняемым кодом в привычном смысле. Это описание интерфейса. Во время сборки и запуска WPF превращает это описание в реальные объекты: `Grid`, `TextBlock`, `Button`, `TabControl` и так далее.

### Схема 2. Как работает Binding

```text
TextBlock в XAML
  ↓
Text="{Binding OperatorName}"
  ↓
WPF смотрит DataContext
  ↓
находит свойство OperatorName
  ↓
берёт значение
  ↓
показывает текст на экране
```

Если `DataContext` не назначен, Binding не знает, где искать свойство.

### Схема 3. Как нажимается кнопка ответа

```text
Пользователь нажал кнопку ответа
  ↓
Button.Command
  ↓
SelectAnswerCommand
  ↓
RelayCommand.Execute(...)
  ↓
MainViewModel.SelectAnswer(...)
  ↓
удаляются лишние шаги после текущего
  ↓
добавляется следующий шаг
  ↓
ObservableCollection уведомляет WPF
  ↓
ItemsControl перерисовывает список
```

Это центральная логика интерактивного сценария.

### Схема 4. Почему `ObservableCollection`, а не `List`

```text
List<T>
  ↓
может хранить элементы
  ↓
но не сообщает WPF, что список изменился

ObservableCollection<T>
  ↓
может хранить элементы
  ↓
сообщает WPF: элемент добавлен / удалён / список изменился
  ↓
UI обновляется автоматически
```

### Схема 5. Где живёт логика

```text
View / XAML
  отвечает за внешний вид

Code-behind
  только стартовая связка окна и минимальная инфраструктура

ViewModel
  состояние экрана и команды

Service
  получение сценария и бизнес-операции

Model
  простые данные: шаг, ответ, идентификатор следующего шага
```

Такое разделение не усложняет проект ради красоты. Оно нужно, чтобы каждую часть можно было понимать и проверять отдельно.

# ПРИНЦИП «ПОЧЕМУ НЕ ТАК»: АНТИПАТТЕРНЫ В ЭТОМ ПРОЕКТЕ

## Коротко

Начинающий часто пишет работающий, но трудно развиваемый код. В этом разделе показано, какие решения допустимы в первом рабочем срезе, но не должны остаться в финальной архитектуре.

## Подробно

### Антипаттерн 1. Весь сценарий в `MainWindow.xaml.cs`

Плохо:

```csharp
private void YesButton_Click(object sender, RoutedEventArgs e)
{
    ScriptPanel.Children.Add(new TextBlock
    {
        Text = "Следующий текст сценария..."
    });
}
```

Почему плохо:

- окно начинает знать текст сценария;
- окно начинает знать правила переходов;
- окно само создаёт элементы интерфейса;
- невозможно нормально тестировать логику переходов;
- при росте сценария файл станет огромным.

Лучше:

```text
ScriptService хранит сценарий.
MainViewModel выбирает следующий шаг.
MainWindow.xaml только показывает VisibleSteps.
```

### Антипаттерн 2. Создавать кнопки вручную в C#

Плохо:

```csharp
var button = new Button();
button.Content = answer.Text;
button.Click += Answer_Click;
AnswersPanel.Children.Add(button);
```

Почему плохо:

- UI создаётся не декларативно;
- труднее менять внешний вид;
- сложнее использовать стили;
- логика и разметка смешиваются.

Лучше:

```xml
<ItemsControl ItemsSource="{Binding Answers}">
    <ItemsControl.ItemTemplate>
        <DataTemplate>
            <Button Content="{Binding Text}" Command="{Binding SelectCommand}" />
        </DataTemplate>
    </ItemsControl.ItemTemplate>
</ItemsControl>
```

### Антипаттерн 3. Хранить только текст без идентификаторов шагов

Плохо:

```csharp
new ScriptAnswer("Да", "Следующий текст")
```

Почему плохо:

- нельзя переиспользовать шаг;
- трудно строить ветвления;
- трудно проверять сценарий;
- текст становится логическим ключом, а это ненадёжно.

Лучше:

```csharp
new ScriptAnswer
{
    Text = "Да",
    NextStepId = "presentation_offer"
};
```

Идентификатор шага — это стабильная техническая ссылка. Текст можно поменять, а `NextStepId` оставить прежним.

### Антипаттерн 4. Заменить `ObservableCollection` на `List`

Плохо:

```csharp
public List<VisibleStepViewModel> VisibleSteps { get; } = new();
```

Почему плохо: WPF не узнает, что ты добавил или удалил шаг. Список изменится в памяти, но интерфейс может не обновиться.

Лучше:

```csharp
public ObservableCollection<VisibleStepViewModel> VisibleSteps { get; } = new();
```

### Антипаттерн 5. Использовать `Click` вместо команд в финальной версии

В первом уроке `Click` допустим, потому что он проще. Но в финальной версии лучше `ICommand`.

Почему:

- команда живёт во ViewModel;
- её можно тестировать отдельно;
- View не содержит бизнес-логики;
- проще отключать кнопку через `CanExecute`.

# УСИЛЕНИЕ УРОКОВ: ДОБАВЬ ЭТИ ПРОВЕРКИ ПОСЛЕ КАЖДОГО СЛОЯ

## Коротко

Ниже — дополнительные задания и проверки для каждого урока. Они не заменяют основной материал, а добавляют учебную глубину.

## Урок 1. Минимальное окно

### Что проверить

```text
1. Проект создаётся командой dotnet new wpf.
2. Приложение запускается командой dotnet run.
3. Окно появляется без ошибок.
```

### Мини-задания

1. Измени заголовок окна.
2. Измени ширину и высоту окна.
3. Найди в проекте `App.xaml` и `MainWindow.xaml`.
4. Объясни письменно, чем отличается `.xaml` от `.xaml.cs`.

### Типичная ошибка

Ошибка:

```text
The name InitializeComponent does not exist in the current context
```

Частая причина: нарушена связь между `MainWindow.xaml` и `MainWindow.xaml.cs`, например не совпадает `x:Class`.

## Урок 2. Вкладки

### Что проверить

```text
1. Есть шесть вкладок.
2. Каждая вкладка переключается.
3. В каждой вкладке есть временный текст.
```

### Мини-задания

1. Добавь вкладку `Справка`.
2. Временно добавь в неё `TextBlock` с текстом `Раздел в разработке`.
3. Удали вкладку и убедись, что приложение всё ещё собирается.

### Почему это важно

`TabControl` — хороший пример контейнера. Он не просто показывает текст. Он управляет набором вкладок и текущим выбранным содержимым.

## Урок 3. Стартовый диалог

### Что проверить

```text
1. При запуске появляется отдельное окно.
2. В нём есть два поля.
3. Кнопка «Сохранить» закрывает окно.
4. Пустые значения не ломают программу.
```

### Мини-задания

1. Добавь третье поле `Номер линии`.
2. Пока не подключай его к сценарию.
3. Просто проверь, что его можно ввести и прочитать.

### Почему это важно

Диалог — это отдельный `Window`. В WPF окно не обязано быть главным. Можно создавать вспомогательные окна для ввода настроек, подтверждения действия или показа подробностей.

## Урок 4. Подключение диалога к главному окну

### Что проверить

```text
1. Главное окно получает имя оператора.
2. Главное окно получает имя клиента.
3. Эти данные отображаются на экране или подставляются в текст.
```

### Мини-задания

1. Если оператор не ввёл имя, подставь `Оператор`.
2. Если клиент не введён, подставь `Клиент`.
3. Убери лишние пробелы через `Trim()`.

### Почему это важно

Это первый маленький пример передачи данных между окнами. Позже такая же идея будет применяться между `View` и `ViewModel`.

## Урок 5. Первый интерактивный шаг

### Что проверить

```text
1. На экране есть первый текст сценария.
2. Есть кнопки ответов.
3. Нажатие на кнопку добавляет следующий видимый блок.
```

### Мини-задания

1. Добавь ответ `Не уверен`.
2. Сделай для него отдельный временный следующий текст.
3. Нажми сначала `Да`, потом вернись и нажми `Не уверен`, если такая возможность уже реализована.

### Почему это важно

На этом этапе логика ещё может быть простой. Главное — доказать, что интерактивная цепочка вообще возможна.

## Урок 6. Модели

### Что проверить

```text
1. Есть модель ответа.
2. Есть модель шага.
3. Один шаг содержит список ответов.
4. Ответ может ссылаться на следующий шаг через NextStepId.
```

### Мини-задания

1. Добавь свойство `Order` для порядка шага.
2. Не используй его в UI.
3. Просто пойми, что модель может хранить не только текст, но и технические данные.

### Почему это важно

Модель — это не элемент интерфейса. Модель отвечает на вопрос: **какие данные существуют в предметной области?** В нашем случае предметная область — сценарий оператора.

## Урок 7. Сервис сценария

### Что проверить

```text
1. Сценарий больше не хранится в MainWindow.
2. Сценарий создаётся в ScriptService.
3. ViewModel или окно получает готовые шаги из сервиса.
```

### Мини-задания

1. Добавь метод `GetFirstStepId()`.
2. Верни из него идентификатор первого шага.
3. Используй этот метод вместо строкового литерала в ViewModel.

### Почему это важно

Сервис отделяет получение сценария от отображения сценария. Позже сценарий можно будет читать из JSON, базы данных или файла, а интерфейс менять почти не придётся.

## Урок 8. ObservableCollection

### Что проверить

```text
1. Видимые шаги хранятся в ObservableCollection.
2. Добавление шага обновляет экран.
3. Удаление шага обновляет экран.
```

### Мини-задания

1. Временно замени `ObservableCollection` на `List`.
2. Посмотри, что изменится.
3. Верни `ObservableCollection` обратно.

### Почему это важно

WPF не умеет читать мысли. Он обновляет интерфейс тогда, когда объект сообщает об изменении. `ObservableCollection` как раз сообщает об изменениях списка.

## Урок 9. MainViewModel

### Что проверить

```text
1. В MainWindow стало меньше логики.
2. Состояние хранится в MainViewModel.
3. DataContext указывает на MainViewModel.
```

### Мини-задания

1. Добавь свойство `WindowTitle` во ViewModel.
2. Привяжи заголовок окна к этому свойству.
3. Измени значение и проверь, что заголовок меняется.

### Почему это важно

ViewModel — это не «ещё один класс ради паттерна». Это место, где экран получает своё состояние и действия.

## Урок 10. Команды

### Что проверить

```text
1. Кнопка ответа вызывает ICommand.
2. В code-behind нет обработчика Click для ответа.
3. Логика выбора ответа находится во ViewModel.
```

### Мини-задания

1. Добавь команду `RestartCommand`.
2. Пока пусть она только очищает список шагов и добавляет первый шаг.
3. Добавь кнопку `Начать заново`.

### Почему это важно

Команда — мост между кнопкой и логикой. В MVVM кнопка не должна знать, как именно устроена логика сценария.

## Урок 11. Binding

### Что проверить

```text
1. MainWindow.xaml показывает данные из ViewModel.
2. ItemsControl показывает VisibleSteps.
3. Кнопки ответов создаются через DataTemplate.
```

### Мини-задания

1. Переименуй свойство в ViewModel и посмотри, что сломается.
2. Верни правильное имя.
3. Посмотри окно Output в Visual Studio на ошибки Binding.

### Почему это важно

Binding — центральный механизм WPF. Если его не понять, WPF будет казаться магией. На самом деле WPF просто берёт имя свойства из XAML и ищет это свойство в `DataContext`.

## Урок 12. Финальная структура

### Что проверить

```text
1. Проект собирается.
2. Проект запускается.
3. Есть папки Models, Services, ViewModels.
4. Сценарий работает.
5. Выбор раннего ответа скрывает поздние шаги.
```

### Мини-задания

1. Добавь новый шаг сценария.
2. Добавь новый ответ, ведущий к этому шагу.
3. Проверь, что ранний выбор удаляет лишнюю ветку.

### Почему это важно

Финальная структура должна быть не просто красивой, а проверяемой. Если ты можешь добавить новый шаг без переписывания окна, значит архитектура выбрана правильно.

# УГЛУБЛЯЮЩИЙ БЛОК: ЧЕГО НЕ ХВАТАЕТ ДО ПОЛНОЦЕННОГО WPF-УРОВНЯ

## Коротко

Для первого WPF-проекта не нужно знать весь WPF. Но чтобы расти дальше, нужно понимать несколько фундаментальных механизмов. Этот блок не заменяет отдельные уроки, а показывает, что изучать следующим этапом.

## 1. DependencyProperty — свойство зависимости

### Коротко

`DependencyProperty` — это специальный тип свойства в WPF. Он нужен для Binding, стилей, анимаций, наследования значений и шаблонов.

### Подробно

Обычное C#-свойство выглядит так:

```csharp
public string Title { get; set; } = "";
```

WPF может работать и с обычными свойствами, особенно во ViewModel. Но сами элементы WPF используют более мощную систему свойств. Например, `TextBlock.Text`, `Button.Content`, `FrameworkElement.Width`, `Control.Background` — это свойства, которые участвуют в WPF property system.

Зачем это нужно:

- Binding может устанавливать значение свойства;
- Style может устанавливать значение свойства;
- Animation может плавно изменять значение свойства;
- значение может наследоваться по дереву элементов;
- WPF может эффективно хранить значения тысяч элементов.

Для начинающего важно не писать свои `DependencyProperty` сразу, а понимать, почему WPF-свойства ведут себя не как обычные C#-свойства.

Официальная опора: Microsoft Learn описывает WPF property system и dependency properties как расширение обычной системы свойств .NET для возможностей WPF.

## 2. RoutedEvent — маршрутизируемое событие

### Коротко

`RoutedEvent` — событие, которое может проходить через дерево элементов, а не обрабатываться только на том элементе, где оно возникло.

### Подробно

Обычное событие в C# обычно принадлежит конкретному объекту:

```csharp
button.Click += Button_Click;
```

В WPF многие события могут «маршрутизироваться» по дереву элементов. Например, клик возник на `Button`, но обработчик может быть назначен выше — на контейнере.

Это связано с тем, что WPF-интерфейс — дерево:

```text
Window
  Grid
    StackPanel
      Button
```

Маршрутизируемые события особенно важны для:

- сложных шаблонов контролов;
- обработки событий на уровне контейнера;
- командной системы;
- пользовательских элементов управления.

Для нашего первого проекта глубокое использование `RoutedEvent` не требуется, но `RoutedEventArgs e` в обработчиках ты уже видишь. Это аргументы маршрутизируемого события.

## 3. ResourceDictionary — словарь ресурсов

### Коротко

`ResourceDictionary` — место, где WPF хранит переиспользуемые ресурсы: цвета, кисти, стили, шаблоны, размеры.

### Подробно

Если цвет кнопки задавать прямо в каждой кнопке, получится дублирование:

```xml
<Button Background="LightGreen" />
<Button Background="LightGreen" />
<Button Background="LightGreen" />
```

Лучше вынести общий стиль:

```xml
<Window.Resources>
    <Style TargetType="Button">
        <Setter Property="Margin" Value="4" />
        <Setter Property="Padding" Value="8,4" />
    </Style>
</Window.Resources>
```

Теперь все кнопки в окне получают общий внешний вид. Это особенно важно для твоего ТЗ, где нужно визуально приблизиться к скриншотам.

## 4. Style — стиль

### Коротко

`Style` — набор значений свойств, который можно применить к одному или нескольким элементам.

### Подробно

Стиль отвечает на вопрос: **как должен выглядеть элемент?**

Например:

```xml
<Style x:Key="AnswerButtonStyle" TargetType="Button">
    <Setter Property="Margin" Value="4" />
    <Setter Property="Padding" Value="10,6" />
    <Setter Property="MinWidth" Value="80" />
</Style>
```

Использование:

```xml
<Button Style="{StaticResource AnswerButtonStyle}" Content="Да" />
```

Так мы отделяем внешний вид от структуры интерфейса.

## 5. ControlTemplate — шаблон элемента управления

### Коротко

`ControlTemplate` определяет, из каких визуальных частей состоит элемент управления.

### Подробно

`Style` меняет свойства существующего контрола. `ControlTemplate` позволяет глубже изменить его внешний вид.

Например, кнопка в WPF — это не просто прямоугольник. У неё есть шаблон: граница, фон, содержимое, визуальные состояния. Если нужен совсем другой вид кнопки, меняют шаблон.

Для первого проекта лучше не начинать с `ControlTemplate`. Сначала достаточно `Style`. Но когда потребуется точно повторить дизайн скриншотов, шаблоны могут понадобиться.

## 6. DataTemplate — шаблон данных

### Коротко

`DataTemplate` отвечает на вопрос: **как показать объект данных на экране?**

### Подробно

В нашем проекте есть объект:

```csharp
VisibleStepViewModel
```

Сам по себе он не имеет внешнего вида. WPF не знает, как его красиво показать. Поэтому мы даём `DataTemplate`:

```xml
<DataTemplate>
    <Border>
        <StackPanel>
            <TextBlock Text="{Binding Text}" />
            <ItemsControl ItemsSource="{Binding Answers}" />
        </StackPanel>
    </Border>
</DataTemplate>
```

Теперь каждый объект `VisibleStepViewModel` превращается в карточку шага.

## 7. Dispatcher — диспетчер UI-потока

### Коротко

`Dispatcher` — механизм WPF, который управляет выполнением кода в UI-потоке.

### Подробно

WPF-интерфейс должен обновляться из UI-потока. Если позже ты добавишь HTTP-запрос, чтение файла или таймер в другом потоке, нельзя будет просто так менять UI из фонового потока.

Упрощённо:

```text
Фоновая операция получила данные
  ↓
нужно обновить UI
  ↓
обновление должно попасть в Dispatcher UI-потока
```

На первом этапе мы избегаем сложной многопоточности. Но для реального приложения это важно.

## 8. Async/await — асинхронность

### Коротко

`async/await` позволяет не блокировать интерфейс во время долгих операций.

### Подробно

Если программа читает файл, обращается к API или загружает сценарий, UI не должен зависать. Поэтому долгие операции делают асинхронными.

Плохо:

```csharp
Thread.Sleep(5000);
```

Пользователь видит зависшее окно.

Лучше:

```csharp
await Task.Delay(5000);
```

UI-поток не блокируется.

Для нашего проекта асинхронность понадобится позже, если сценарий будет загружаться из файла, базы или API.

## 9. Validation — валидация ввода

### Коротко

Валидация — проверка данных пользователя.

### Подробно

В стартовом диалоге пользователь вводит имя оператора и имя клиента. На первом слое можно просто подставлять значения по умолчанию. В более зрелой версии лучше добавить правила:

- имя оператора не пустое;
- имя клиента не пустое;
- нельзя вводить только пробелы;
- слишком длинные значения обрезаются или показывают ошибку.

В WPF есть несколько подходов к валидации:

- простая проверка в коде кнопки;
- проверка во ViewModel;
- `INotifyDataErrorInfo`;
- validation rules в Binding.

Для начинающего лучше начать с простой проверки во ViewModel или в коде диалога, а потом изучить полноценную WPF-валидацию.

## 10. Тестирование логики сценария

### Коротко

UI тестировать сложно. Логику переходов тестировать проще, если она вынесена из окна.

### Подробно

Вот почему мы выносили сценарий в модели, сервис и ViewModel. Можно проверить:

```text
Если открыт шаг A и выбран ответ Да, должен появиться шаг B.
Если после A уже были B и C, но пользователь выбрал другой ответ в A, B и C должны исчезнуть.
Если NextStepId пустой, новый шаг не добавляется.
```

Такой тест можно написать без запуска окна. Это признак нормальной архитектуры.
